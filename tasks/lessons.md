# lessons

## direction resets

- 사용자가 `이 디자인은 다른 사람이나 회사에 못 준다`고 말하는 순간은 보통 microcopy 문제가 아니라 product framing mismatch다. 그 시점에는 wording thin slice를 더 열지 말고, 먼저 `AGENTS / master brief / decision log` 에서 shell contract 자체를 다시 고정하는 편이 맞다.
- 원하는 방향이 `ERP/company shell + visible AI roles + meeting/workday cues` 인데 repo truth가 여전히 `display-only HQ metaphor` 나 `office-first 금지`에 묶여 있으면, UI는 계속 themed website처럼 보일 수밖에 없다. 이런 경우에는 visual polish보다 source-of-truth reset이 먼저다.
- 외부 reference가 둘 이상일 때는 역할을 분리해서 써야 한다. 이번처럼 `claw-empire` 는 shell/product posture, `OpenHarness` 는 harness/governance robustness reference로 나눠 두면 한 레포의 장점을 다른 축에 과하게 투사하는 실수를 줄이기 쉽다.
- 디자인 방향을 장기적으로 흔들리지 않게 하려면 repo root에 agent-readable `DESIGN.md` 를 두는 편이 좋다. 이번처럼 `awesome-design-md` 패턴을 참고해 palette, typography, component rules, do/don'ts, responsive behavior를 먼저 고정해 두면 이후 UI slice가 매번 제품 방향부터 다시 토론하지 않아도 된다.
- 레퍼런스 레포를 많이 모았을수록 더 중요한 건 분류다. `shell posture`, `design-system discipline`, `harness governance`, `automation`, `debate flow` 를 분리해 읽지 않으면 한 레포의 강점이 다른 축으로 새어 들어와 scope drift를 만들기 쉽다.
- company-shell 리셋의 첫 배치는 inner surface를 바로 갈아엎기보다 `masthead -> summary -> nav` 같은 global shell frame부터 다시 잡는 편이 맞다. 바깥 프레임이 여전히 poster/website rhythm이면 `Mission / Council / Execution / Deliverables` 안쪽을 아무리 회사식으로 바꿔도 전체 제품이 여전히 데모 사이트처럼 읽히기 쉽다.
- visual reset에서도 runtime hook과 query selector를 지키는 편이 smoke와 browser harness를 안정적으로 유지했다. 이번처럼 `summary-card-*`, `refresh-*`, `data-surface`, `nav-button` 같은 anchor를 유지한 채 shell chrome만 다시 설계하면 ERP 톤을 크게 바꾸면서도 runtime/gate 회귀 위험을 낮출 수 있다.
- global shell을 ERP 톤으로 바꾼 뒤 `Mission` 첫 화면이 여전히 `회의실 hero` 로 시작하면 intake semantics와 meeting semantics가 겹쳐 보여 첫 인상이 다시 흐려진다. `Mission` 은 먼저 `등록대장 / 배정 / 다음 처리 트리거` 로 읽히게 만들고, 실제 회의실 연출은 `Council` 로 넘기는 편이 surface authority를 더 분명하게 만든다.
- `Mission` 리디자인에서는 top hero와 intake desk만 바꾸고 `mission row rail`, task-link wiring, approval/action route를 건드리지 않는 편이 좋았다. 이렇게 하면 회사형 intake board 톤을 크게 바꾸면서도 `smoke-ui-slice-*` 와 `smoke-qa-slice-07` 회귀 범위를 좁게 유지할 수 있다.
- `Council` 을 실제 회의실처럼 보이게 만들려면 hero 한 블록만 바꾸면 부족했다. `회의 운영 데스크 / 참석 등록부 / 오늘 회의 안건 / 권고안 선반 / 승인 선반` 같은 top-frame을 열었다면, 같은 batch 안에서 viewport strip, empty-state, right narrative deck도 모두 `참석 기록 / 권고 / 승인` 문법으로 같이 맞춰야 surface 전체가 half-reset이 아니라 하나의 meeting room으로 읽힌다.
- `Execution` 을 회사형 control surface로 바꿀 때도 같은 규칙이 통했다. `작업 지시 보드` 만 바꾸고 left narrative, gate deck, empty-state, mission/council handoff copy를 예전 `작전실` 문법에 남겨 두면 surface authority가 다시 갈라졌다. `작업 지시 / 승인 게이트 큐 / 최근 실행 로그 / 차단 사유 / 실행 준비 패킷` 을 같은 batch 안에서 같이 맞춰야 work-order surface가 한 제품 흐름으로 읽힌다.
- `Deliverables` 도 같은 규칙을 따라야 했다. top summary만 `delivery desk` 로 바꾸고 left packet shelf, right approval/close-out desk, pending/no-task language를 예전 `결과 보고` 문법에 남겨 두면 surface가 다시 반쪽짜리로 읽혔다. `결과 패킷 / 리뷰 라인 / 승인 라인 / 종료 보고 데스크 / 전달 패킷 선반` 을 같은 batch 안에서 같이 맞춰야 delivery surface authority가 유지된다.
- structure slice에서 helper를 adapter 밖으로 추출할 때는 파일 하나만 옮기고 끝내면 안 된다. 이번처럼 `openai-responses-retry-policy.js` 로 retry/backoff governance를 분리했다면, 같은 batch 안에서 builder-preflight code-context allowlist와 관련 smoke path도 같이 갱신해야 prompt boundary와 실제 실행 boundary가 다시 어긋나지 않는다.
- 구조 강화 배치에서는 generic 이름을 먼저 크게 열기보다 provider-scoped helper 이름으로 시작하는 편이 안전했다. 실제 reuse가 확인되기 전에는 `provider-retry-policy` 같은 넓은 이름보다 `openai-responses-retry-policy` 처럼 boundary를 드러내는 이름이 scope drift를 막기 쉽다.
- 회사형 shell로 방향을 틀었는데도 first viewport가 여전히 웹사이트처럼 보인다면, surface 내부 카드를 더 손보기 전에 `chrome density` 를 먼저 다시 잡아야 한다. 특히 `left rail`, `operations topbar`, `summary register strip` 이 marketing hero 리듬을 갖고 있으면 나머지 `Mission / Council / Execution / Deliverables` 가 아무리 회사형이어도 전체 제품 인상은 다시 데모 사이트 쪽으로 기운다.
- company shell을 더 product처럼 보이게 만들 때는 `2열 카드형 sidebar nav` 보다 `single-column row nav` 가 훨씬 사내 툴에 가깝게 읽혔다. 좌측 레일에서 각 surface를 작은 desk row처럼 세워 두면 first viewport가 즉시 ERP/workspace 쪽으로 기운다.
- shell KPI도 큰 rounded card로 두면 다시 데모 사이트처럼 보였다. 같은 정보라도 `head strip + register body` 구조로 눌러 주면 current project, runtime, queue, gate가 훨씬 더 내부 운영 도구처럼 읽혔다.
- shell frame을 바꾼 뒤에도 `surface focus strip` 과 `company pulse strip` 이 여전히 소개 카드처럼 남아 있으면 first viewport 인상이 다시 웹사이트 쪽으로 기운다. shared helper를 `현재 등록 / 현재 분장 / 다음 이동` 같은 register vocabulary로 바꾸고, surface별 본문에 들어가기 전에 업무 대장처럼 읽히게 만드는 편이 회사형 톤을 더 안정적으로 끌어올렸다.
- shell strip만 등록대장처럼 바꾸고 body list가 여전히 세로 카드 묶음이면 제품 인상이 다시 데모 사이트로 떨어졌다. `Taskboard / Logs / Artifacts / Decision Inbox` 같은 반복 목록은 공통 `등록 / 현재 상태 / 다음 처리` 열 구조를 직접 넣고, row 자체를 flatter register로 눌러 주는 편이 회사형 ERP 톤을 훨씬 더 빠르게 올렸다.
- shell row를 조밀하게 눌렀는데도 first viewport가 여전히 데모처럼 보인다면, body를 더 만지기 전에 `department board` 같은 회사형 overview surface를 한 번 더 넣는 편이 빠르다. summary 바로 아래에 `본부 접수 / 회의실 / 실행 셀 / 보고 데스크 / 승인선` 현황판을 두고, 각 칸을 `현재 상태 / 다음 처리 / 데스크 열기` register로 읽히게 만들자 제품이 훨씬 더 사내 ERP처럼 보였다.
- 회사형 톤으로 밀고 싶다면 serif/editorial heading은 오래 붙잡지 않는 편이 좋았다. shell chrome과 board title, register title이 계속 serif로 남아 있으면 아무리 구조를 ERP처럼 바꿔도 인상이 다시 curated demo 쪽으로 기운다. first viewport 핵심 heading부터 sans 기반으로 통일하는 편이 더 빠르게 제품 인상을 바꿨다.
- shell 구조를 ERP처럼 옮긴 뒤에도 여전히 데모처럼 보이면 다음 타겟은 새 보드가 아니라 공용 chrome 값이다. sidebar, summary, register, active-nav, detail-action 같은 shared surface의 gradient, shadow, radius를 한 단계 더 말려 주는 편이 새 회사 장식을 더 붙이는 것보다 훨씬 빨리 `내부 운영 도구` 인상을 만든다.
- shell chrome을 한 번 말린 뒤에도 first viewport가 여전히 showpiece처럼 보이면, 다음 타겟은 `surface lead / viewport handoff / entry panel` 에 남아 있는 radial gradient와 showcase glow다. 이 구간을 `flat board + thin top rail + restrained shadow` 로 한 단계 더 누르자 `Mission / Council / Execution / Deliverables` 가 훨씬 더 사내 ERP 화면처럼 읽혔다.
- shell 구조와 copy를 ERP처럼 바꿔도 `dark showcase panel`, `hero glow`, `cartoon avatar ornament` 가 surface header에 남아 있으면 제품 인상은 다시 game/demo 쪽으로 미끄러진다. 이런 단계에서는 새 설명을 더 쓰기보다 ornament layer를 실제로 꺼 두고, 그 제거 상태 자체를 smoke로 고정하는 편이 visual drift를 훨씬 잘 막았다.
- left rail이 브랜드 소개만 담고 있으면 shell 전체를 아무리 눌러도 첫인상이 여전히 랜딩 페이지처럼 남는다. sidebar에 `현재 프로젝트 / 현재 데스크 / 열린 실행 / 열린 승인선` 같은 실제 운영값 register를 올리자, 같은 레이아웃 안에서도 훨씬 더 사내 업무 도구처럼 읽혔다.

## core loop

- runtime core는 file-based thin slice로 닫는 접근이 유효했다.
- gate enforcement는 UI보다 먼저 닫는 편이 구조적으로 안정적이었다.
- flags 모델(`blocked / waitingApproval / waitingDecision`)은 상태 확장보다 단순하고 유지보수에 유리했다.
- planner -> architect -> task-breaker -> builder preflight를 runtime/coordinator 위에 얇게 쌓는 방식은 broad refactor 없이 core loop를 닫는 데 유효했다.
- reviewer provenance는 task 전체의 latest artifact 조합이 아니라 latest builder live-mutation bundle 하나에 고정해야 drift를 막을 수 있었다.
- commit-package readiness와 local commit readiness도 latest passing reviewer bundle 하나에 고정하는 편이 provenance와 duplicate 제어를 단순하게 유지했다.
- release-package readiness는 task 전역 latest artifact를 다시 조합하지 말고 latest successful local commit bundle 하나에 고정해야 post-commit provenance drift를 막을 수 있었다.
- close-out readiness는 release-package readiness를 재사용하지 말고 별도 계산으로 두는 편이 승인 완료 release bundle과 terminal Done duplicate를 함께 다루기에 안정적이었다.

## shell / gate handling

- 초기 UI는 write action보다 gate/status visibility에 집중하는 편이 pack 제약과 구조 안정성에 더 맞았다.
- local browser smoke가 `listen EPERM 127.0.0.1`로 실패하면 product regression으로 바로 해석하지 말고, 먼저 sandbox port-binding 제한인지 분리해서 outside-sandbox로 재실행하는 편이 원인 분리가 빨랐다.
- shell server 안의 최소 mutation route + snapshot echo만으로도 write UI를 generic API layer 없이 붙일 수 있었다.
- linked worktree selection도 같은 패턴으로 닫을 수 있었고, detection은 current project_path에서 server-derived로 계산하고 release/close-out guard ownership은 coordinator에 그대로 두는 편이 drift를 막았다.
- detected linked worktree와 registered project 매핑은 raw string path가 아니라 canonical realpath 기준으로 처리해야 symlink나 alias path가 섞여도 shell switch가 흔들리지 않았다.
- active project를 detected linked worktree root로 바꿀 때는 project migration을 시도하지 말고 기존 create/select 흐름만 재사용하면서 selected task/run/artifact/inbox selection을 새 project scope로 reset/hydrate 하는 편이 범위를 안정적으로 유지했다.
- linked worktree create도 Project Bootstrap의 단일 write action과 server-side `git worktree add` route 하나로 닫는 편이 범위와 UX를 안정적으로 유지했다.
- linked worktree create 직후에는 task migration이나 `task.worktreeRef` auto-set을 붙이지 말고, canonical realpath 기준 project register/select 재사용까지만 닫는 편이 기존 relation model과 core loop semantics를 흔들지 않았다.
- pending inbox action route 하나와 snapshot 기반 selection 갱신만으로 human gate loop를 닫을 수 있었다.
- approval authorization은 최신 approval record만 보는 것으로는 부족하고, 해당 record가 최신 preflight 또는 commit-package target을 가리키는지까지 확인해야 stale allow를 막을 수 있었다.
- live mutation, reviewer, commit-package, local commit enablement는 UI 추정보다 runtime/coordinator readiness를 그대로 읽는 편이 drift를 줄였다.
- `project_path`처럼 operator가 바로 고칠 수 있는 execution prerequisite는 pending decision item으로 승격하기보다 coordinator-derived readiness summary와 shell blocked copy로 드러내는 편이 더 안전했다. 그래야 경로를 고친 직후 planner -> architect -> task-breaker -> builder-preflight entry가 자동으로 풀리고, 사람이 별도 unblock action을 하지 않아도 shell truth가 runtime truth와 함께 움직인다.
- provider opt-in도 같은 원칙으로 project-level coarse readiness summary만 노출하고, project.readiness 자체는 provider health로 재정의하지 않는 편이 경계를 안정적으로 유지했다.
- explicit live opt-in은 env 존재만으로 auto-enable하지 말고, config/env/readiness가 하나라도 비정상이면 local-stub fallback 없이 fail-closed로 막는 편이 no-secret-leak과 operator intent를 함께 지키기 쉬웠다.
- 첫 concrete live provider는 broad capability 비교보다 현재 `execute(request) -> outputText` 계약과 planner artifact path에 가장 작은 delta로 붙는 대상을 고르는 편이 범위를 안정적으로 유지했다.
- first live adapter canonical id를 `openai-responses`로 고정하고 `live-provider`는 임시 alias로만 두는 편이 이후 provider drift와 문서/상태 불일치를 줄이기 쉽다.
- planner live output은 Responses Structured Outputs(`text.format.type=json_schema`)로 `artifactMarkdown + normalizedResult`를 함께 받는 편이 기존 artifact contract와 decision routing을 가장 작게 유지한다.
- `outputText`는 `response.output_text`를 우선 사용하고, 없으면 `output` array의 `output_text` content를 aggregate하는 편이 응답 shape 변화에 더 안전하며 fixed-index 접근보다 유지보수가 쉽다.
- actual live adapter도 project-level readiness는 planner 기준 coarse summary만 노출하고, non-planner role은 adapter readiness에서 `degraded/blocked`로 닫는 편이 UI 개편 없이 scope를 유지하기 쉽다.
- malformed live provider response는 run error로만 남기고 artifact를 만들지 않는 편이 fail-closed와 provenance cleanliness를 함께 지키기 쉽다.
- first live scope는 planner-only로 고정하고, concrete model default는 repo에 박지 말고 operator-pinned project config로 두는 편이 v1 semantics와 provider drift를 함께 막기 쉬웠다.
- architect live 확장은 coordinator 쪽 broad refactor보다 explicit anchor exact-match 검증과 adapter-rendered canonical markdown으로 닫는 편이 downstream boundary semantics를 안정적으로 유지하기 쉽다.
- task-breaker live처럼 upstream artifact가 둘 이상인 단계는 latest artifact를 타입별로 독립 선택하지 말고 `latest plan + matching latest architecture` provenance chain 자체를 anchor로 강제하는 편이 breakdown drift를 막기 쉽다.
- task-breaker provenance mismatch는 provider 호출 뒤에 정리하기보다 coordinator/runtime guard에서 먼저 막는 편이 stale breakdown artifact와 misleading live logs를 함께 줄이기 쉽다.
- 기존 UI/parser contract가 이미 고정된 artifact(`breakdown` 등)는 live structured output을 raw markdown 대체로 밀어넣기보다 adapter-rendered canonical markdown으로 닫는 편이 UI 변경 없이 범위를 유지하기 쉽다.
- task-breaker live browser smoke도 project-level provider summary는 coarse readiness만 브라우저에서 보고, 실제 `builder | human gate` 의미론은 `/api/snapshot`, artifact content, run logs로 닫는 편이 UI drift를 줄이기 쉽다.
- prompt contract의 allowed handoff와 escalation wording은 coordinator가 실제 허용하는 `nextStage` 집합과 일치시켜야 fail-closed noise와 문서 drift를 줄이기 쉽다.
- builder-preflight live가 구현된 뒤에는 docs/pack/todo의 design-only wording을 즉시 정리해야 다음 slice 논의가 stale premise 위에서 흔들리지 않는다.
- builder live mutation live는 preflight/approval pair만으로는 부족하고 `plan + architecture + breakdown + preflight + approval + target allowlist + baseline digests`까지 포함한 exact-match anchor로 잠그는 편이 mutation provenance를 안정적으로 유지하기 쉽다.
- builder live mutation bundle은 `change-summary / patch / diff`를 따로 남기기보다 atomic bundle로 다루고, validation 실패 시 repo restore + artifact 미생성 + approval 미소비로 닫는 편이 provenance cleanliness와 retry semantics를 함께 지키기 쉽다.
- reviewer live를 열지 않는 slice에서는 live-mode project의 reviewer blocked/degraded 상태를 silent fallback이 아니라 explicit operator step으로 문서화하는 편이 scope drift를 줄이기 쉽다.
- reviewer live를 여는 slice에서는 latest task artifact를 다시 조합하지 말고 latest successful builder live-mutation bundle 하나만 anchor로 잡은 뒤, canonical review markdown 검증이 끝난 다음에만 review artifact를 저장하는 편이 fail-closed provenance를 유지하기 쉽다.

## smoke / fixtures

- localhost 기반 UI smoke는 sandbox listen 제약을 받으므로 승인 가능한 로컬 서버 smoke로 정리하는 편이 실용적이었다.
- limited live mutation smoke는 repo root 파일을 직접 fixture로 쓰면 기존 mutation marker가 누적되어 no-op write를 숨길 수 있으므로, clean fixture project를 따로 만들어야 안정적이다.
- synthetic downstream bundle smoke는 reviewer, commit-package, local commit의 provenance 규칙을 빠르게 검증하는 데 유효했지만, upstream live-mutation path를 대체할 수는 없었다.
- release-package처럼 범위를 좁힌 후속 slice는 기존 dev-loop smoke를 흔들기보다 별도 execution smoke로 duplicate/stale/blocked 케이스를 닫는 편이 회귀 추적에 유리했다.
- close-out도 같은 패턴으로 synthetic downstream execution smoke를 따로 두는 편이 release-package 회귀와 Done transition 회귀를 분리해서 추적하기 좋았다.
- release gate 근거로는 synthetic smoke만으로 부족하고, planner부터 local commit까지 한 번에 도는 real-path dev loop smoke가 필요했다.
- local-stub live mutation smoke fixture는 단일 placeholder 파일만 두는 것으로는 불안정할 수 있었고, current allowlist selection에 맞는 `prompts/builder.md` 같은 실제 target 후보를 함께 만들어 두는 편이 primary-shell CTA smoke를 더 안정적으로 닫는 데 유리했다.
- stale smoke assertion은 과거 route-specific 에러 문구보다 현재 runtime/coordinator guard를 source of truth로 따라가는 편이 유지보수에 유리했다.
- browser click smoke는 DOM에서 landing/selection/visibility만 보고, task/worktree/close-out 의미론은 `/api/snapshot`과 artifact API로 확인하는 편이 범위와 brittleness를 함께 줄였다.
- provider opt-in browser smoke도 같은 원칙으로 DOM에서는 summary, readiness, allowed/reason, disabled state만 확인하고 fail-closed 의미론은 API로 한 번 더 닫는 편이 안정적이었다.
- planner plus architect live browser smoke도 provider config mutation 자체는 API로 두고, 브라우저에서는 opt-in 반영 상태와 planner/architect click-through만 확인하는 편이 현재 shell 구조에서 더 안정적이었다.
- qa-slice-04처럼 architect live browser smoke를 추가할 때도 project-level provider summary는 coarse readiness만 브라우저에서 보고, planner/architect ready와 downstream degraded는 direct coordinator assertion으로 닫는 편이 DEC-032 경계와 UI 안정성을 함께 지키기 쉽다.
- human-gate architect smoke는 Decision Inbox surface 노출만 브라우저에서 확인하고, `kind / sourceType / blocksTask / task flags`는 `/api/snapshot`으로 닫는 편이 selector brittleness 없이 의미론을 유지하기 쉽다.
- synthetic reviewed-bundle browser smoke도 current reviewer anchor contract를 그대로 따라야 했다. builder live-mutation fixture에는 `preflightArtifactId`, `preflightRunId`, `inputArtifactIds`, `inputRunIds`, approval target linkage, `changedFiles`, 그리고 change-summary의 base64 `File Updates`가 모두 맞아야 reviewer readiness drift를 막을 수 있었다.
- Task Detail help copy는 upstream artifact id가 아직 없을 수 있으므로 null-safe로 렌더링해야 browser smoke 전에 shell이 죽지 않는다.
- Playwright CLI 세션 고정은 wrapper 전용 env var에 기대지 말고 각 호출에 `--session`을 명시하는 편이 로컬 재현성과 디버깅 안정성이 높았다.

- mission form panel density도 같은 원칙이 유지됐고, create-form 전체는 copy를 더 줄이기보다 `task-create-form-compact / field-grid-compact / field-compact / form-actions-compact` 같은 modifier class로 spacing만 줄이는 편이 semantics drift 없이 scanability를 가장 안전하게 높였다.

- mission detail header density도 같은 원칙이 유지됐고, selected mission context와 current state entry도 새 copy를 만들기보다 `panel-header-tight / panel-copy-tight / token-row-compact`를 재사용해 상단 gap만 줄이는 편이 drift 없이 scanability를 가장 안전하게 높였다.
- mission title strip density도 같은 원칙이 유지됐고, selected mission identity와 bounded goal도 새 상태나 helper copy를 늘리기보다 `relation-strip-compact / card-title-row-tight / detail-copy-compact` 같은 modifier class로 strip 밀도만 줄이는 편이 가장 안전했다.
- detail polish가 title strip까지 닫히면, 그 다음에는 microcopy slice를 계속 쪼개기보다 acceptance freeze로 전환해 현재 `Mission / Council / Execution / Deliverables` baseline을 completion target으로 고정하는 편이 drift를 훨씬 줄였다.
- acceptance freeze 단계에서는 새 capability를 더 열기보다 existing runtime fixture와 source-based acceptance smoke로 “이 baseline이 완성 상태다”를 고정하는 편이 가장 안전했다. completion 이후 남는 optional real-live/provider housekeeping은 반드시 non-blocking 운영 트랙으로 분리해야 했다.
- completion baseline이 frozen 된 뒤에는 새 primary-shell slice를 계속 열기보다 남은 작업을 기존 non-blocking housekeeping backlog로 되돌리는 편이 drift를 가장 잘 막았다. baseline completion과 운영 housekeeping을 같은 queue에 두면 다시 product blocker처럼 보이기 쉽다.
- 상위 방향성 항목(`ai-orchestration-pivot-v2`)이 실제로 구현된 뒤에도 backlog에 `[ ]`로 남겨두면 이후 housekeeping이 다시 product blocker처럼 보인다. completion freeze 이후에는 그 방향성 항목 자체를 완료로 닫고, 남은 작업만 explicit non-blocking 또는 later `vNext` entry로 되돌려야 ledger drift를 막기 쉽다.
- deferred/rejected section도 같은 원칙이 적용됐고, docs가 이미 `deferred` 또는 `rejected`로 못 박은 항목을 todo 끝에 `- [ ]`로 남겨두면 실제 open implementation item처럼 오해되기 쉽다. 그런 항목은 unchecked backlog가 아니라 명시적 deferred/rejected note로 닫아 두는 편이 ledger drift를 가장 잘 막았다.
- todo에 active unchecked item이 0개가 된 시점은 그냥 상태가 아니라 baseline contract라서, `rg` 확인만 하고 끝내기보다 zero-open 상태 자체를 smoke로 고정해 두는 편이 이후 stale checkbox 재유입을 가장 빨리 잡아냈다.
- zero-open backlog가 곧 clean worktree를 뜻하지는 않으므로, cleanup decision 없이 바로 정리하려 들기보다 현재 dirty state를 known baseline smoke로 먼저 고정하는 편이 accidental drift와 cumulative intended changes를 가장 잘 분리했다.
- dirty worktree를 실제로 정리할 시점이 오더라도 파일 하나씩 ad-hoc으로 고르기보다 `policy/contracts`, `runtime/retention`, `shell/ui`, `browser coverage`, `docs/smokes`처럼 stable bucket inventory를 먼저 고정해 두는 편이 staging 실수와 cleanup drift를 줄이기 쉬웠다.
- cleanup order도 같은 원칙이 적용됐고, bucket inventory를 만든 뒤에는 `policy/contracts -> runtime/retention -> shell/ui -> browser coverage -> docs/smokes`처럼 stage order까지 먼저 고정해 두는 편이 cross-bucket staging bleed와 불필요한 partial commit 결정을 줄이기 쉬웠다.
- PR merge와 branch cleanup까지 끝난 뒤에도 usable baseline을 가정만 하지 말고, clean `main`에서 representative source/runtime/browser smoke를 다시 돌려 merged baseline이 실제로 살아 있는지 한 번 더 닫는 편이 가장 안전했다.
- roadmap 하단이나 backlog 쪽에서 어떤 방향을 이미 `implemented on current main`으로 닫았다면, 문서 상단 phase summary의 `next product step` wording도 같은 턴에 같이 정리해야 stale future-scope drift를 막기 쉽다.
- optional real-live verification이 non-blocking인 이상, 현재 Codex execution context에서 `launchctl getenv`가 비어 있으면 그것도 운영 현실로 취급하고 representative live smoke를 다시 돌려 fresh `skipped_missing_env` evidence로 닫는 편이 stale env 추정보다 훨씬 안전했다.
- completed mission list framing도 같은 원칙이 유지됐고, row scanability를 높일 때는 새 `mission.status`나 extra runtime field를 만들기보다 linked task `Done` + saved close-out artifact 기준으로 `active / completed` framing과 next-cycle-ready copy만 얹는 편이 drift를 막기 가장 쉬웠다.
- council preview도 같은 원칙이 유지됐고, `Mission`에서 recommendation과 alignment를 더 잘 보이게 만들 때는 별도 mission shadow field를 만들지 않고 existing `councilSession.summary / recommendation / selectedPlan / alignment`만 재조합하는 편이 drift를 막기 가장 쉬웠다.
- active mission detail의 preview가 여러 strip로 불어나기 시작하면 정보를 더 추가하기보다 `Council / Execution / Deliverables / Current Best Next Step`를 하나의 compact snapshot으로 접는 편이 scanability를 훨씬 빨리 회복시켰다. 이때도 기존 preview helper를 재사용하면 runtime semantics를 건드리지 않고 density만 줄일 수 있었다.
- compact snapshot으로 접은 뒤에도 사용자는 "그래서 어디로 가야 하는가"를 바로 묻게 되므로, 각 preview 줄에 owning surface와 handoff copy를 붙이는 편이 별도 CTA를 늘리지 않고도 navigation intent를 훨씬 명확하게 만들었다.
- mission list row는 detail보다 훨씬 빨리 훑는 표면이라서, row에는 모든 provenance를 다시 늘어놓기보다 `한 줄 상태 요약 + 한 줄 next owner + 최소 token`만 남기는 편이 실제 scanability를 크게 높였다. 자세한 provenance는 selected mission detail에 두는 편이 더 낫다.
- selected mission detail도 같은 원칙이 적용됐고, helper copy는 상태 설명을 반복하기보다 surface 역할과 다음 읽기 경로만 짧게 남기는 편이 훨씬 읽기 쉬웠다. copy tighten은 structure나 semantics를 바꾸지 않고도 제품 감각을 빠르게 올릴 수 있었다.
- browser smoke도 같은 원칙을 따라야 했고, first-run shell이 `Mission`으로 바뀐 뒤에는 기존 `Taskboard` bootstrap selector를 계속 기대하기보다 `Mission -> Start With This Project -> Taskboard` handoff를 명시적으로 밟는 편이 stale regression을 가장 작게 정리했다.
- provider opt-in browser smoke도 같은 원칙이 유지됐고, project-level provider summary는 계속 `Taskboard`에서 coarse readiness만 확인하되, first-run 진입 자체는 `Mission`에서 시작한 뒤 명시적으로 `Taskboard`로 handoff하는 편이 미션-first shell 이후 stale bootstrap expectation을 가장 작게 정리했다.
- retention도 같은 원칙이 유지됐고, Tier A/B/C rules를 이미 정규화해 둔 상태라면 첫 slice는 곧바로 delete/archive/GC를 여는 것보다 explicit operator invocation이 필요한 inspect-only retention consumer preview를 먼저 두는 편이 provenance drift를 가장 작게 막았다.
- retention apply도 같은 원칙이 유지됐고, destructive action을 한 번에 넓히기보다 `archive -> delete -> gc`를 explicit 상태 전이로 나누고 Tier A는 fail-closed로 막아 둔 편이 inspectability와 provenance safety를 함께 유지하기 가장 쉬웠다.
- execution preview도 같은 원칙이 유지됐고, `Mission`에서 latest stage / current gate / blocked reason을 더 잘 보이게 만들 때는 새 mission execution state를 만들기보다 existing `run`, `approval bridge`, `inbox`, `task.flags`를 조합하는 편이 drift를 막기 가장 쉬웠다.
- deliverables preview도 같은 원칙이 유지됐고, `Mission`에서 latest artifact / review / approval 상태를 더 잘 보이게 만들 때는 새 mission delivery state를 만들기보다 existing artifact ordering, review status, approval summary를 그대로 재조합하는 편이 drift를 막기 가장 쉬웠다.
- current best next-step framing도 같은 원칙이 유지됐고, `Mission`에서 무엇을 눌러야 하는지 더 직접적으로 보여주려면 새 action graph를 만들기보다 existing council / execution / deliverables / completion preview를 우선순위만 정해 조합하는 편이 drift를 막기 가장 쉬웠다.
- 사용자가 `Council`을 캐릭터 군처럼 느끼길 원할 때는 helper copy나 participant count만으로는 부족했고, existing `participants + transcript`를 재사용한 repeatable cast-card layer를 `Mission Detail`과 `Council`에 같이 깔아야 실제로 “역할군이 보인다”는 체감이 생겼다. 이때도 새 runtime role이나 shadow field를 만들지 않는 편이 가장 안전했다.
- 오케스트레이션 기본 언어를 한국어로 유지하려면 UI literal만 번역해서는 부족했고, `decision log + runtime-generated council copy + representative smoke expectation`을 같은 slice에서 함께 고정해야 영어가 source truth처럼 다시 스며들지 않았다. 내부 role id나 runtime key는 그대로 두고 visible copy만 한국어로 정규화하는 편이 blast radius를 가장 작게 유지했다.
- 같은 한국어화 slice 안에서도 raw runtime status를 그대로 token이나 header에 꽂아 두면 다시 영어가 새어 나오기 쉬웠다. `Mission / Council / Execution / Deliverables`처럼 여러 표면이 같은 상태값을 공유할 때는 `display helper`를 먼저 만들고 그 helper를 대표 shell 전체에 같이 연결해야 drift를 가장 빨리 막을 수 있었다.
- 같은 원칙이 `Taskboard / Logs / Artifacts / Decision Inbox`와 project-level operator bootstrap에도 그대로 적용됐다. primary shell만 한국어화해 두고 guard panel, inbox action status, provider bootstrap form을 raw runtime/status 문자열에 맡겨 두면 secondary surface로 내려가는 순간 영어가 다시 새어 나왔고, 이 경우에도 새 상태를 만들기보다 display helper와 visible panel copy를 같이 묶어 고정하는 편이 drift를 가장 빨리 막았다.
- 같은 한국어화 작업에서도 panel 겉면만 번역하고 relation strip, artifact preview heading, guard reason, builder live mutation approval, package status token을 raw 문자열로 남겨 두면 내부 preview에서 다시 영어가 새어 나왔다. 이 영역은 slice를 더 잘게 쪼개기보다 `reason/mode/verdict/package/run/provider readiness` display helper를 먼저 만들고 preview/guard/bootstrap 내부에 같이 연결하는 편이 drift를 가장 빨리 막았다.
- 같은 원칙이 linked worktree와 operator error에도 적용됐다. panel headline만 번역해 두고 `submit/run` 경로의 thrown error, refresh status, `task.worktreeRef` 안내문, runtime validation message를 영어로 남겨 두면 실제 사용 순간 다시 영어가 튀어나왔고, 이 경우에도 UI action handler와 runtime create validation을 같은 slice에서 같이 한국어화하는 편이 drift를 가장 빨리 막았다.
- 사용자가 `Council`을 캐릭터 군이 아니라 직급이 있는 참모진으로 읽길 원하면, 단순 role label만 나열하는 카드로는 부족했다. 같은 네 role id를 유지하더라도 `지휘 서열 / 직급 / 담당 임무`를 display metadata로 같이 고정하고 선임 카드에만 시각적 무게를 더하는 편이 기획 의도와 실제 체감 사이의 간격을 가장 빨리 줄였다.
- 사용자가 처음 접속했을 때 회사 본부와 회의실 같은 인상을 원하면, 기존 운영형 패널 위에 카드를 조금 더 얹는 정도로는 부족했다. `Mission`과 `Council` 상단을 별도 `브리핑실 + 회의 테이블 + 착석한 참모진` 구조로 재프레이밍하고, 입력 행위도 `미션 생성`보다 `안건 올리기`처럼 회의 시작 행위로 보이게 바꾸는 편이 제품 의도를 훨씬 빠르게 이해시켰다.
- 첫 화면만 회의실처럼 보이고 하위 섹션이 다시 `상세 / 추천안 / 실행` 운영 패널 어휘로 돌아가면 사용자는 흐름을 잃는다. 같은 slice 안에서 `안건 브리프 -> 회의 결론 -> 작전 지휘실`처럼 하위 헤더와 helper copy도 같은 회사/회의 서사로 이어 줘야 “AI들이 회의하고 지시를 내린다”는 제품 감각이 유지됐다.
- 같은 회사/회의 서사를 더 밀고 싶다면 `Deliverables / Logs / Artifacts / Decision Inbox`도 각각 `결과 보고실 / 관제실 로그실 / 증적 보관실 / 관제실 결재함`처럼 공간 이름을 먼저 줘야 했다. 첫 화면만 본부처럼 보이고 아래 표면이 다시 generic 운영 명사로 돌아가면 사용자는 제품 지도를 잃었고, 반대로 방 이름과 보고/결재 흐름을 붙이면 전체가 하나의 본부 동선처럼 읽혔다.
- 입력 표면은 연출보다 단순성이 먼저였다. `Mission` 첫 구역에서 회의실 서사를 너무 많이 반복하면 사용자는 어디에 입력해야 하는지 잠깐 멈췄고, `안건 접수 데스크`처럼 제목·목표·선택 경계만 남긴 한 덩어리 접수판으로 줄이자 실제 오더 지점이 훨씬 빨리 읽혔다.
- 입력 데스크를 줄인 뒤에도 `Mission` 목록 row와 `안건 브리프`가 계속 장문이면 첫 화면은 여전히 느리게 읽힌다. 이 경우 row에는 `현재 상태 + 다음 행동 + 최소 token`만 남기고, 브리프는 `핵심 4줄`처럼 짧은 요약판으로 접는 편이 실제 오더 UX를 훨씬 빠르게 만들었다.
- 같은 원칙이 `Taskboard`에도 그대로 적용됐다. 작업판은 세부 제어가 많아도 되지만 첫 화면만큼은 `현재 셀 / 다음 행동 / 승인선`과 간단한 `실행 셀 접수`만 먼저 보여 주는 편이, 긴 detail block을 처음부터 다 펼쳐 두는 것보다 훨씬 빨리 읽혔다.
- `Taskboard` 상세도 첫 블록에서 곧바로 run/worktree/provenance를 모두 나열하면 operator가 다시 멈춘다. 상세를 열었을 때는 `현재 상태 / 막힌 이유 / 다음 실행`만 먼저 판단판으로 보여 주고, run/worktree/가드/증적은 아래 세부 블록으로 남기는 편이 실제 제어 속도를 더 높였다.
- 같은 원칙이 `관제실 로그실 / 증적 보관실 / 관제실 결재함` 상세에도 그대로 적용됐다. operator가 이 표면까지 내려왔더라도 첫 블록에서 raw log, relation strip, 원문 markdown, 결재 액션을 전부 한꺼번에 읽게 하기보다 `현재 상태 / 핵심 이유 / 다음 확인` 판단판을 먼저 두는 편이 실제 확인 속도를 더 높였다.
- 판단판을 올린 뒤에도 그 바로 아래 메타 block이 길면 첫 화면은 다시 무거워진다. `run 기본 정보 / 증적 기본 정보 / 결재 기본 정보`처럼 첫 메타 묶음도 compact token + 짧은 kv-grid로 접고, raw log·원문 markdown·결재 액션은 그 아래로 미루는 편이 실제 스캔 속도를 더 높였다.
- 같은 원칙으로 raw 섹션 앞 helper copy도 너무 길면 첫 화면이 다시 늘어진다. `보고 연결선 / 보고 미리보기 / 보관 원문 / 처리 메모` 앞 안내 문구는 짧은 한 줄로 줄이거나 제거하고, raw viewer 높이도 compact class로 낮추는 편이 실제 관제 화면의 first-screen density를 더 잘 지켰다.
- 사용자가 `사무실에 직급별 AI 캐릭터가 앉아 있는 느낌`을 원할 때는 단순 회의실 카드와 직급 텍스트만으로는 부족했다. 같은 네 role id를 유지하더라도 `데스크 이름 / 자리 성격 / 착석 상태 / 직급`을 한 덩어리 station으로 묶어 보여 주는 편이 훨씬 빨리 캐릭터 존재감과 조직 구조를 이해시켰다.
- 그 다음 단계에서는 station만으로도 아직 문서형 UI처럼 보일 수 있었다. 외부 이미지 없이도 `아바타 실루엣 / 배지 / 데스크 소품` 같은 display-only layer를 station과 cast 카드에 같이 넣어 주면, runtime 계약을 건드리지 않고도 각 AI가 자기 자리에서 일하는 캐릭터처럼 훨씬 빨리 읽혔다.
- 캐릭터가 보여도 이용 흐름이 함께 보이지 않으면 첫 방문자는 여전히 멈춘다. `귀여운 얼굴 + 역할별 소품 + 4단 onboarding strip`을 같은 첫 화면에 같이 두면, 제품 성격과 사용 순서를 동시에 이해시킬 수 있어 가독성이 훨씬 좋아졌다.
- 외부 agent product를 참고할 때도 `회사 연출` 자체를 복제하면 금방 조잡해진다. 실제로는 `목표 가시성 / 역할 가시성 / 흐름 가시화 / 운영 기준`만 뽑아 `charter`로 먼저 보여 주는 편이, HQ 톤을 유지하면서도 누구나 바로 이해하는 orchestration shell을 만드는 데 더 유효했다.
- `charter`만 있어도 여전히 "지금 어느 단계고 누가 맡는가"가 한 번에 안 읽힐 수 있었다. `dify`의 workflow readability와 `crewAI`의 crews + flows framing을 참고해 단계별 `owner + status + next summary`를 같은 rail에 묶자, HQ 연출을 유지한 채 first-view 이해 속도가 훨씬 좋아졌다.
- UI가 `display-only HQ/crew metaphor`까지 가면 source-of-truth docs도 그 경계를 같이 고정해야 했다. 그냥 `office-first 금지` 문구만 남겨 두면 구현과 계약이 계속 충돌하므로, `company simulator는 여전히 금지`이되 `orientation layer로서의 crew/HQ/flow ownership`은 허용된다는 식으로 boundary를 더 정밀하게 적는 편이 drift를 가장 빨리 줄였다.
- 흐름 rail이 아무리 잘 보여도 read-only면 첫 화면에서 다시 아래 패널을 찾게 된다. `owner + status + next summary` rail을 그대로 `Mission / Council / Execution / Deliverables` 이동 버튼으로 연결하자, operator는 설명을 읽은 직후 같은 시야 안에서 바로 다음 표면으로 점프할 수 있어 control surface로서의 가치가 훨씬 커졌다.
- 첫 viewport rail만 clickable이어도 목록 row가 여전히 read-only 요약이면 다시 스크롤과 탐색이 생긴다. `Mission` row 자체에 `안건 / 회의 / 실행 / 보고` quick rail을 같이 두자, 사용자는 목록에서 바로 mission을 고르고 해당 owning surface로 이동할 수 있어 실제 제어 속도가 한 단계 더 빨라졌다.
- 목록 row까지 빨라져도 오른쪽 `안건 브리프`가 긴 설명으로 먼저 시작하면 다시 판단이 늦어진다. 우측 패널 첫 블록을 `현재 판단 / 바로 이동 / 이유` 3단 control deck로 바꾸자, 사용자는 현재 상태와 다음 표면을 먼저 이해한 뒤 필요할 때만 아래의 `브리프 핵심 4줄 / 브리프 액션`으로 내려가게 됐다.
- 이 원칙은 `Mission`에만 한정되지 않았다. `회의 결론 / 작전 지휘 메모 / 최신 보고 현황`도 긴 설명이나 세부 블록으로 먼저 시작하면 다시 속도가 떨어졌고, 각 우측 패널 상단을 같은 `현재 판단 / 바로 이동 / 이유` control deck으로 통일하자 표면이 달라도 읽는 방식이 같아져 훨씬 덜 조잡하게 느껴졌다.
- 우측 패널만 판단 우선으로 바꾸고 좌측 목록이 여전히 id-first row면 사용자는 다시 목록에서 멈춘다. `관제실 로그실 / 증적 보관실 / 관제실 결재함` 목록도 `현재 상태 / 다음 확인` 두 줄 control row로 맞추자, advanced-ops에서도 선택 단계부터 같은 읽기 규칙을 유지할 수 있었다.
- 목록과 판단판을 줄여도 표면 상단에 `header + notice`가 따로 쌓여 있으면 first viewport가 다시 무거워진다. `작업판 / 관제실` 계열은 상단 orientation을 한 번만 보여 주는 compact lead strip로 합치는 편이, 같은 안내를 유지하면서도 실제 스캔 속도를 더 높였다.
- 오른쪽 패널만 `현재 판단 / 바로 이동 / 이유` 구조로 압축하고 왼쪽은 여전히 header + provenance strip를 먼저 보여 주면 표면마다 읽는 규칙이 달라져 다시 조잡하게 느껴진다. `작전실 / 결과 보고실` 왼쪽도 같은 판단 우선 deck로 맞추고, 깊은 relation strip은 그 아래로 내리는 편이 first viewport 가독성을 더 안정적으로 맞춰 줬다.
- `Mission / Council`처럼 hero 연출이 강한 표면은 hero 아래의 좌우 authority를 짧은 handoff strip으로 먼저 선언하는 편이 좋았다. 브리핑실 연출, 왼쪽 desk/record, 오른쪽 judgment가 모두 각자 설명을 시작하면 사용자가 다시 멈추므로, `왼쪽은 무엇을 다루고 오른쪽은 어디서 결론을 내리는지`를 한 줄 규칙으로 먼저 고정하는 편이 first viewport 이해 속도를 더 빠르게 올렸다.
- operator 표면도 같은 원칙이 통했다. `Taskboard`처럼 hero가 없는 화면이라도 `left lane / right detail / 지금 먼저 갈 곳`을 먼저 고정하면 사용자가 deck과 레인에 들어가기 전에 읽는 규칙을 바로 이해할 수 있었고, 기존 lead strip보다 실제 조작 속도가 더 빨랐다.
- queue형 관제 표면도 예외가 아니었다. `Decision Inbox`처럼 `대기 큐 + 최근 처리 + 우측 상세`가 동시에 있는 화면은 상단 summary만으로는 시선이 분산되기 쉬웠고, `왼쪽 큐 / 오른쪽 판단 / 지금 열기`를 먼저 박아 두는 편이 실제로 어떤 열을 먼저 읽어야 하는지 훨씬 빨리 이해시켰다.
- run 원문을 다루는 표면도 같은 규칙을 따라야 했다. `관제실 로그실`은 원문 자체가 무겁기 때문에 더더욱 `왼쪽 목록 / 오른쪽 판단 / 지금 열기`를 먼저 고정해야 했고, 그래야 사용자가 raw log에 들어가기 전에 먼저 어떤 run을 고르고 어디로 되돌아갈지 빠르게 이해할 수 있었다.
- provenance-heavy 표면은 log보다 더 강하게 같은 규칙이 필요했다. `증적 보관실`은 타입, 연결선, structured preview, raw source가 한 화면에 몰리기 쉬워서, `왼쪽 목록 / 오른쪽 판단 / 지금 열기`를 먼저 선언하지 않으면 사용자가 어디서 선택하고 어디서 판단하는지 금방 놓쳤다.
- advanced-ops 표면에서 `지금 열기`는 단순히 owning cell로 되돌리는 버튼이 아니라 현재 human gate를 가장 먼저 가리켜야 했다. `관제실 로그실 / 증적 보관실`처럼 deep surface에 내려온 뒤에도 task가 이미 `waitingApproval` 또는 pending inbox 상태라면, `영향 셀`보다 `관제실 결재함`을 먼저 열어 주는 편이 실제 조작 흐름을 훨씬 덜 끊었다.
- queue형 표면에서는 handoff strip만 줄여도 충분하지 않았다. `관제실 결재함`처럼 deck, 대기 큐, 최근 처리 열이 연달아 붙는 화면은 queue 헤더도 `무엇을 고르는 열인지`만 짧게 말하고 count token으로 역할을 분리해야 첫 viewport에서 같은 설명이 두 번 시작되지 않았다.
- `Taskboard`처럼 상단에 strip, deck, 접수 폼이 연달아 붙는 화면은 각 블록의 역할을 더 강하게 나눠야 했다. strip과 deck이 이미 `현재 셀 / 다음 행동 / 승인선`을 설명하고 있다면, 접수 폼은 새 셀 생성 규칙만 짧게 말하고 제어 흐름 설명은 더 얹지 않는 편이 first viewport를 훨씬 빨리 읽히게 했다.
- handoff strip이 이미 표면의 읽는 규칙을 설명하고 있다면, 그 바로 아래 deck은 surface 소개를 반복하지 않는 편이 좋았다. `관제실 로그실`처럼 raw 정보가 무거운 표면은 strip을 먼저 보여 주고, deck은 `현재 run / 다음 확인 / 현재 맥락`처럼 실제 판단 카드만 남겨야 first viewport가 훨씬 덜 무겁게 읽혔다.
- 같은 규칙은 provenance-heavy surface에도 그대로 적용됐다. `증적 보관실`도 handoff strip이 이미 `왼쪽 목록 / 오른쪽 판단 / 지금 열기`를 설명하고 있으므로, deck까지 `보관실 소개`를 반복하기보다 `현재 증적 / 다음 확인 / 현재 맥락`만 남겨야 first viewport가 더 빨리 읽혔다.
- queue형 surface도 결국 같은 정리가 필요했다. `관제실 결재함`은 queue header가 이미 `지금 처리 / 감사 추적` 역할을 설명하고 있으므로, deck은 `현재 안건 / 다음 확인 / 현재 맥락`처럼 현재 판단 카드만 남기고 strip 뒤에 붙이는 편이 first viewport를 더 일정하게 읽히게 했다.
- 홈 first viewport를 더 "제품답게" 보이게 만들고 싶을 때도 새 semantics를 늘리는 게 답은 아니었다. `directive strip + live signal pod + mode dock`처럼 기존 `project / runtime / run / gate / surface` 데이터만 관제판 문법으로 재배열하자, office theater를 다시 늘리지 않고도 control-plane 톤과 scanability를 함께 올리기 쉬웠다.
- 홈 도크가 제품적으로 보여도 바로 아래 본문이 곧장 예전 상세 패널 톤으로 떨어지면 여전히 끊겨 보였다. nav 아래에 `현재 도크 / 현재 포커스 / 지금 체크` bridge strip을 한 번 더 두고, 사람이 개입해야 하는 순간엔 여기서 바로 `결정함`을 먼저 가리키게 하자 홈 control-plane과 active surface body가 훨씬 덜 단절돼 보였다.
- 홈 shell이 구조상 잘 이어져도 active tone이 늘 같은 베이지 상태면 다시 평평해졌다. `현재 surface` 값을 그대로 써서 dock active state와 focus strip accent만 바꿔도, 새 semantics 없이 `Mission / Council / Execution / Deliverables / Taskboard / Logs / Artifacts / Decision Inbox`가 각자 다른 표정으로 읽혀 훨씬 덜 템플릿처럼 느껴졌다.
- 홈에서 accent가 살아도 실제 표면 첫 블록이 다시 중립색으로 시작하면 흐름이 또 끊겼다. 각 surface의 맨 위 `hero / handoff strip / top deck`에만 `surface-entry-frame`을 붙이고 active surface accent를 읽게 하자, 아래 deep detail까지 건드리지 않고도 홈 control-plane 톤이 본문 entrance까지 자연스럽게 이어졌다.
- accent가 hero와 deck에서만 멈추면 여전히 "보는 화면" 같고 "조작하는 화면" 같지는 않았다. 같은 surface tone을 첫 viewport의 CTA, order desk, focused input border까지만 얇게 이어 주자, 새 semantics 없이도 조작감이 훨씬 더 제품답게 살아났고 deep ops 패널까지 과하게 물들이지 않을 수 있었다.
- 결재 surface는 accent를 단순히 카드와 버튼에만 주는 것으로는 부족했고, 실제 처리 권한이 모여 있는 행 자체를 하나의 action shelf로 묶어야 했다. `Decision Inbox`처럼 승인/반려/해결이 핵심인 화면은 `처리 라벨 + 짧은 handling copy + 액션 버튼`을 한 묶음으로 정리했을 때 훨씬 덜 평평하고 더 바로 쓰는 화면처럼 느껴졌다.
- helper-heavy surface도 같은 규칙이 필요했다. `Artifacts`처럼 provenance 요약과 helper panel이 길게 이어지는 화면은 action row를 그냥 inline button으로 두면 다시 문서처럼 보이므로, `guard-summary`와 preselected approval hint 안의 조작 줄도 별도 shelf로 묶어야 실제 handling zone이 더 또렷하게 읽혔다.
- stage-control surface는 action shelf를 더 강하게 요구했다. `Execution`처럼 승인, 라이브 변경, 리뷰, 커밋, 릴리스, 종료 정리 CTA가 한 column에 차례로 쌓이는 화면은 각 줄을 그냥 동일한 button row로 두면 단계감이 흐려졌고, relation strip 안의 각 action row를 shelf처럼 묶어 주는 편이 지금 어느 단계의 handoff를 처리하는지 훨씬 빨리 읽히게 했다.
- report-heavy surface도 마지막 handoff row가 평평하면 전체 흐름이 다시 끊겼다. `Deliverables`처럼 요약과 보고가 중심인 화면은 `실행으로 이동`, `다음 사이클 시작`, `관제실 직행` 같은 후속 CTA 줄까지 같은 shelf 톤으로 묶어야 보고실에서 실제 다음 행동으로 넘어가는 감각이 더 자연스럽게 이어졌다.
- action shelf가 여러 surface로 늘어나면 각 곳을 따로 예쁘게 꾸미는 것보다 `상단 accent line / row gap / help copy 폭` 같은 리듬 규칙을 하나로 맞추는 편이 훨씬 덜 조각나 보였다. `Decision Inbox / Artifacts / Execution / Deliverables`는 맥락은 달라도 handling zone의 박자는 같아야, 전체 control plane이 한 제품처럼 읽힌다.
- action shelf의 문장이 길어지면 다시 설명 패널처럼 느껴졌다. 같은 shelf 안에서는 먼저 `여기서 무엇을 처리하고 다음 어디로 이어지는지`만 한 문장으로 고정하고, 더 깊은 제약이나 provenance는 아래 block에 남겨 두는 편이 실제 조작 속도를 더 잘 살렸다.
- queue surface는 `다음 확인`과 `다음 처리`를 섞어 쓰면 strip, deck, shelf가 다시 따로 노는 느낌을 줬다. `Decision Inbox`처럼 실제로 사람이 바로 판단하는 표면은 first viewport 전체를 `현재 상태 / 다음 처리` 문법으로 통일하는 편이 훨씬 덜 끊기고 더 즉시 조작 가능하게 읽혔다.
- 관찰형 surface는 반대로 `현재 판단` 같은 넓은 표현보다 대상 객체를 직접 말하는 편이 더 빨리 읽혔다. `Logs`처럼 run이 중심인 화면은 strip, deck, detail 모두 `현재 run / 다음 확인` 문법으로 맞춰 두는 편이 훨씬 덜 추상적이고, 원문 로그를 보기 전 판단 속도도 더 빨랐다.
- provenance surface도 관찰형이지만 대상 객체를 더 직접 불러 주는 편이 좋았다. `Artifacts`처럼 증적이 중심인 화면은 strip, deck, detail 모두 `현재 증적 / 다음 확인` 문법으로 맞춰야 구조 미리보기와 원문을 보기 전에 무엇을 판단하는지 더 빨리 들어왔다.
- 본문 copy를 정리해도 홈 dock와 focus strip이 예전 요약 문장을 쓰고 있으면 첫인상에서 다시 언어가 갈라졌다. surface별 문법이 생기면 `home -> surface entrance -> body`까지 같은 문장을 재사용해야, control plane 전체가 한 제품처럼 더 자연스럽게 읽혔다.
- 홈 copy도 surface마다 같은 추상도를 써야 했다. `Decision Inbox / Logs / Artifacts`만 객체 중심 문법으로 맞추고 `Mission / Execution / Deliverables`를 예전 요약 말투에 남겨 두면, 홈 dock 한 줄 안에서도 다시 제품 언어가 갈라졌다. `현재 안건 판단 / 바로 이동`, `현재 작전 판단 / 다음 행동`, `현재 보고 판단 / 다음 행동`처럼 같은 수준의 문법으로 맞춰야 control-plane 전체가 덜 템플릿처럼 읽혔다.
- 홈만 맞추고 `Mission` hero 안의 flow rail을 예전 문구로 남겨 두면 바로 다음 viewport에서 다시 말투가 끊겼다. `dock -> focus strip -> charter flow`처럼 연달아 붙는 요약 레이어는 같은 grammar family를 써야, 첫 화면 전체가 하나의 control-plane 문장처럼 읽혔다.
- `Mission` hero 안에서도 support copy가 길어지면 다시 설명 패널처럼 느껴졌다. `목표 헌장`은 목표를 먼저 고정한다고만 말하고, `운영 기준`은 display-only HQ 연출과 실제 gate semantics의 경계만 짧게 못 박는 편이 first viewport를 훨씬 덜 무겁게 만들었다.
- `Mission` hero는 copy만 줄인다고 충분히 poster처럼 보이지 않았다. `charter-card`에 얇은 상단 accent line, 조금 더 큰 목표 hierarchy, 더 타이트한 flow row spacing을 주자, 새 카드나 새 의미를 만들지 않고도 첫 viewport가 훨씬 덜 평평하고 더 의도된 화면처럼 읽혔다.
- explicit harness execution 다음 단계에서 결과를 더 오래 보이게 해야 한다면 frozen `doctor.summary`나 consumer payload를 다시 넓히기보다 UI server local memory와 snapshot-derived helper 한 겹으로 닫는 편이 안전했다. 이번처럼 `derived.latestHarnessExecution`만 추가하고 `Execution`이 active project와 representative harness를 다시 대조하게 두면 refresh 복구는 얻고, producer contract drift는 막을 수 있다.
- local-only execution evidence를 한 단계 더 확장할 때도 같은 규칙이 유효했다. 최근 이력이 필요하면 artifact persistence나 runtime state schema부터 건드리지 말고, 먼저 server-process local bounded array와 `snapshot derived` register로 닫는 편이 훨씬 안전했다. 이번처럼 `recentHarnessExecutions`를 newest-first로 5건만 유지하면 refresh 비교는 얻고, frozen producer/consumer 계약은 그대로 둘 수 있다.
- `Council` boardroom hero도 같은 원리가 있었다. 테이블과 좌석이 전부 비슷한 무게로 깔리면 회의실이 단순 grid처럼 보였고, 중심 테이블 그림자와 좌석 상단 accent rail, compact seat spacing을 조금만 정리해도 훨씬 더 staged scene처럼 읽혔다. 중요한 건 새 장식이 아니라 중심축과 위계였다.
- `Mission / Council` hero처럼 이미 위계가 잡힌 first viewport는 새 블록을 더 쌓기보다 entrance motion을 아주 얇게 넣는 편이 효과적이었다. staggered rise와 작은 hover lift 정도만 줘도 정적인 polished panel에서 staged control scene 쪽으로 인상이 넘어가지만, `prefers-reduced-motion`를 같이 닫아 두지 않으면 쉽게 과해질 수 있었다.
- hero 아래 progress rail도 정적 안내 칩으로 두면 다시 장식처럼 보였다. `Mission / Council`처럼 이미 같은 flow state를 다른 곳에서 계산하고 있다면, 그 rail도 `완료 / 현재 / 다음` 위계를 바로 읽게 맞춰 주는 편이 훨씬 덜 딱딱하고 더 live control surface처럼 느껴졌다.
- hero rail을 live하게 만들어도 바로 아래 handoff strip 카드가 다시 flat utility card로 떨어지면 first viewport 리듬이 끊겼다. 같은 surface-entry-frame 안에서는 strip 카드도 얇은 top rail, 작은 hover lift, 뒤늦은 entrance delay를 같이 가져가야 `hero -> rail -> strip`이 한 세트처럼 읽혔다.
- hero와 handoff strip까지만 리듬을 맞추고 바로 아래 quick rail 버튼을 예전 유틸 칩처럼 남겨 두면 첫 actionable row에서 다시 힘이 빠졌다. `Mission`처럼 사용자가 곧바로 누르게 되는 rail은 얇은 top rail, 작은 hover lift, `다음` 버튼 emphasis, 선택된 row 기준 staggered entrance까지 이어 줘야 first viewport 전체가 같은 control tempo로 읽혔다.
- quick rail 버튼만 살아 있고 그것을 감싸는 mission row 카드가 다시 평평하면 선택 맥락이 반쯤 끊겨 보였다. 선택된 list row 자체에도 얇은 top rail, hover halo, 조금 더 따뜻한 backdrop을 같이 줘야 사용자가 `지금 보고 있는 미션`과 `바로 누를 rail`을 한 덩어리 control zone으로 읽었다.
- mission row 컨테이너를 살린 뒤에도 head typography가 generic list headline처럼 남아 있으면 윗줄과 아랫줄이 다시 분리돼 보였다. `Mission` row처럼 제목과 상태 토큰이 첫 줄에 같이 붙는 경우엔 제목은 조금 더 크게, 토큰은 더 촘촘한 control pod처럼 정리해야 selection emphasis와 quick rail rhythm이 같은 레이어로 읽혔다.
- head를 정리한 뒤에도 goal, summary, next가 거의 같은 무게로 남아 있으면 미션 row 본문이 다시 하나의 문단처럼 뭉개졌다. `Mission` row 아래 두 줄은 `summary`를 더 조용한 supporting line으로 낮추고 `next`를 더 또렷한 action line으로 세워야, 제목 아래에서도 `goal -> summary -> next` 순서가 빠르게 읽혔다.
- `Mission` row에서 goal 줄 자체가 너무 가볍고 줄높이가 느슨하면 제목 아래 첫 문장이 바로 안 꽂혔다. goal은 약간 더 큰 읽기 크기와 안정적인 2줄 clamp 높이를 주고, header와의 gap도 조금 더 조이면 첫 절반이 훨씬 빨리 읽혔다.

- left rail이 실제 operator queue로 읽히길 원할 때는 rail 폭 안에서 카드 수를 더 늘리기보다, 각 row 안에 `현재 / 담당 / 다음` 같은 고정 scan columns를 넣는 편이 더 효과적이다. 이번처럼 `업무 / 검토` row마다 explicit 3-field meta register를 붙이니 detail panel을 열기 전에도 현재 상태와 다음 이동을 바로 비교할 수 있었고, grouped workspace 구조를 유지한 채 list readability를 가장 작게 개선할 수 있었다.
- `운영` 같은 admin 화면은 회사 구조 보기와 인력 수정 폼을 한 블록에 쌓아두면 무엇이 read-only roster이고 무엇이 mutation editor인지 바로 안 읽힌다. 이번처럼 `범위 선택 -> 현재 배정 현황 -> AI agent 추가 / 역할·데스크 편집` 순서로 분리하고, `전체 / 업무 / 검토 / 운영` scope filter를 두면 기존 grouped workspace와 company directory를 유지한 채로도 admin 도구처럼 훨씬 빨리 읽혔다.
- left rail이 이미 `현재 / 담당 / 다음` register를 갖고 있어도 selected row signal이 약하면 사용자는 결국 center panel을 읽고 나서야 현재 위치를 이해하게 된다. 이번처럼 active row에만 `현재 보기` label, 좌측 accent rail, stronger field contrast, `aria-current`/`data-selection-state` hook을 같이 넣어 두면 grouped workspace 구조를 바꾸지 않고도 queue scan speed와 QA proof 둘 다 안정적으로 올릴 수 있었다.
- `운영` 같은 admin surface에서 create form과 assignment row가 같은 필드 밀도로 붙어 있으면 사용자는 무엇이 새 agent 추가인지, 무엇이 기존 배정 수정인지 빨리 구분하지 못한다. 이번처럼 create path에 `추가 역할 / 배정 desk` preview를 먼저 두고, create/update 모두를 `기본 정보`와 `역할·desk 배정(변경)` 섹션으로 쪼개면 같은 grouped workspace 구조 안에서도 mutation editor의 목적이 훨씬 더 명확해졌다.
- `운영` 편집기가 이미 create/update 분리를 갖고 있어도 read-only roster와 editing row 대비가 약하면 여전히 같은 패널처럼 읽힌다. 이번처럼 roster panel을 muted `readonly` 상태로 두고, edit panel과 각 row에 explicit editing markers(`data-panel-state`, `data-editor-state`, `편집 대상`, `역할·desk 수정`)를 주면 사용자는 무엇이 보는 영역이고 무엇이 수정 영역인지 훨씬 빨리 이해했다.
- grouped workspace의 body 가독성은 새 summary를 더 얹는 것보다 panel/row typography와 padding을 한 단계씩 줄이는 편이 효과적이었다. 이번처럼 `control-overview` panel, queue row, register field, inspector step, ops metric의 font size와 inner padding을 같이 조이면 구조를 바꾸지 않고도 `값 -> 상태 -> 다음 액션` 순서가 더 빨리 눈에 들어왔다.
- `운영` mutation editor가 create/update 분리와 editing contrast를 이미 갖고 있어도, footer가 generic button row로 남아 있으면 사용자는 무엇이 저장되는지 마지막 순간까지 다시 읽어야 했다. 이번처럼 create/edit footer를 `저장 결과 / 저장 범위` action bar로 바꾸고 primary action을 오른쪽에 고정하니, grouped workspace 구조를 유지한 채로도 admin tool의 입력-확정 흐름이 훨씬 더 빨리 읽혔다.
- grouped workspace에서 active card만 더 진하게 만드는 것보다 inactive row를 한 단계 더 약하게 눌러 주는 편이 scan speed 개선에 더 직접적이었다. 이번처럼 `업무 / 검토` idle queue row, sidebar company directory의 non-current seat, `운영` team board의 non-active group, readonly roster row를 같이 muted baseline으로 맞추니 구조를 바꾸지 않고도 현재 선택과 현재 편집 범위가 훨씬 빨리 눈에 들어왔다.
- admin tool 입력 흐름이 느리게 읽힐 때는 `기본 정보 / 역할·desk 변경` 처럼 같은 카드 안에 section을 두 번 나누는 것보다, `현재 배정` 은 muted reference panel로 두고 실제 mutation inputs는 `이름 → 역할 → desk` 한 줄 order로 묶는 편이 훨씬 빨리 읽힌다. 이번처럼 create/edit 둘 다 동일한 field order를 쓰고 footer copy도 `현재 roster와 회사 구조에 반영` 수준으로 줄이면 사용자가 save 전에 무엇이 저장되는지 더 쉽게 예측할 수 있었다.
- grouped workspace가 이미 분리돼 있어도 body register 라벨이 `현재 안건 / 연결 실행 / 현재 근거 / 현재 막힘 / 다음 확인 / 현재 편집 범위` 처럼 길게 남아 있으면 값보다 라벨이 먼저 읽혀 scan speed가 다시 떨어진다. 이번처럼 queue/body register를 `상태 / 담당 / 다음`, `안건 / 실행 / 근거`, `패킷 / 타입 / 다음`, `범위 / 프로젝트 / 승인선 / 인력 / 역할 / desk` 수준으로 줄이고, 같은 턴에 register padding과 type scale까지 같이 조여 주면 구조를 바꾸지 않고도 value-first reading이 훨씬 안정적으로 올라간다.
- grouped workspace가 어느 정도 정리된 뒤에는 panel 구조를 또 바꾸기보다 action/helper 문장을 줄이는 편이 더 직접적으로 먹힌다. 이번처럼 `getControlOverviewCheck()` 의 긴 설명문을 `사람 판단을 먼저 처리합니다 / 차단 원인을 먼저 정리합니다 / 안건 정리 후 회의로 넘깁니다` 수준으로 줄이고, `운영` footer도 `반영 / roster와 회사 구조에 추가합니다 / 배정 저장` 으로 압축하니 사용자는 버튼을 누르기 전 마지막 읽기 비용을 훨씬 덜 들였다.
- helper prose를 줄인 다음 단계에서는 button label도 같이 noun-first로 정리하는 편이 scan speed를 더 직접적으로 올렸다. 이번처럼 `협의회 열기 / 실행 열기 / 산출물 열기 / 아티팩트 열기 / 결정함 열기` 를 `협의회 / 실행 / 산출물 / 아티팩트 / 결정함` 으로 줄이고, deliverables의 긴 CTA도 `실행에서 변경` 수준으로 접으니 grouped workspace 구조를 건드리지 않고도 click intent가 훨씬 빨리 읽혔다.
- noun-first action baseline을 잡은 뒤에도 `... 열기` 잔여 문구가 navigation hint, immediate card, guard CTA에 남아 있으면 다시 shell이 들쭉날쭉하게 읽혔다. 이번처럼 `회의 초안 / 태스크 연결 / 실행 데스크 / 안건 브리프 / 관제실 / 결재함 / 영향 셀 / 태스크 상세 / 커밋 가드 / 종료 가드` 로 후속 진입 라벨까지 함께 접고, 버튼 padding도 한 단계 같이 줄이면 grouped workspace 구조를 바꾸지 않고도 click intent와 scan speed를 더 균일하게 맞출 수 있었다.
- 외부 multi-agent repo를 참고할 때는 그 repo의 runtime semantics나 조직 은유를 통째로 가져오지 말고, 여러 레퍼런스에서 공통으로 보이는 `explicit mode / role / workflow framing` 패턴만 얇게 추출하는 편이 안전했다. 이번처럼 [`oh-my-claudecode`](https://github.com/Yeachan-Heo/oh-my-claudecode)의 orchestration framing, [`gstack`](https://github.com/garrytan/gstack)의 role/playbook entry, [`superpowers`](https://github.com/obra/superpowers)의 `design -> plan -> execute` workflow, [`everything-claude-code`](https://github.com/affaan-m/everything-claude-code)의 research-first system 설명을 직접 확인한 뒤, 우리 repo에는 기능 추가 대신 `업무 / 검토 / 운영` 상단에 `사용 순서`를 보여 주는 `workspace playbook strip` 만 들여오니 frozen local-first shell semantics를 깨지 않으면서도 first-use clarity를 높일 수 있었다.
- grouped workspace를 여러 slice에 걸쳐 줄여 온 뒤에도 viewport-handoff immediate card만 `지금 열기` 같이 다른 어휘를 쓰고 있으면 shell tone이 다시 흔들렸다. 이번처럼 taskboard/logs/artifacts/decision-inbox 공통 immediate card를 `바로`로 맞춰서 이미 쓰고 있던 `바로:` token과 vocabulary를 통일하면, 구조나 동작을 건드리지 않고도 마지막 CTA scan speed를 더 균일하게 만들 수 있었다.
- handoff card 안에서 register는 이미 `다음` 으로 읽히는데 narrative deck만 `바로 이동` 이 남아 있으면 같은 surface 안에서도 CTA grammar가 다시 흔들렸다. 이번처럼 mission brief, council, deliverables deck의 middle card를 모두 `다음` 으로 맞추면, 구조를 바꾸지 않고도 handoff intent가 register와 같은 언어로 읽혀 scan speed를 더 안정적으로 유지할 수 있었다.
- CTA vocabulary를 정리할 때는 card label만 보지 말고 같은 흐름을 설명하는 charter summary까지 같이 훑는 편이 안전했다. 이번처럼 narrative deck의 `바로 이동` 을 `다음` 으로 줄인 뒤 intake flow summary까지 `다음 이동` 으로 맞추지 않으면 shell 어휘가 여전히 두 갈래로 남기 때문에, 마지막 남은 exact string을 grep으로 끝까지 제거하는 편이 vocabulary freeze에 더 적합했다.
- action button freeze 단계에서는 strip CTA만 보지 말고 detail rail 아래의 follow-up buttons까지 같이 봐야 했다. 이번처럼 `회의실 열기 / 회의 초안 만들기 / 실행 셀 만들기 / 고급 운영 모드 열기` 가 남아 있으면 main CTA vocabulary를 줄인 뒤에도 shell이 다시 verbose하게 읽히므로, 마지막 follow-up button도 noun-first로 접어 두는 편이 freeze 직전 scan speed를 더 안정적으로 맞출 수 있었다.
- company-shell redesign이 여러 thin slice로 길어지면, 어느 시점부터는 UI를 더 고치기보다 `freeze candidate` 와 `commit candidate boundary` 를 docs로 먼저 고정하는 편이 낫다. 이번처럼 `docs/10_company-shell-design-reset.md` 에 first viewport/sidebar/workspace/vocabulary contract와 include/exclude bundle boundary, freeze gate command set을 명시하고 source smoke로 고정해 두면 다음 턴부터는 `무엇을 더 바꿀지` 가 아니라 `어디서 멈추고 어떤 묶음으로 commit 할지` 를 훨씬 차분하게 결정할 수 있다.
- dirty worktree 안에서 freeze 후보를 실제 commit candidate로 옮길 때는 category만 적어 두면 staging scope가 다시 흔들리기 쉽다. 이번처럼 `docs/10_company-shell-design-reset.md` 에 exact file inventory와 explicit exclusions, `git add .` 금지 staging rule까지 적고 smoke로 고정해 두면 다음 단계에서 unrelated provider/runtime 변경이 섞여 들어갈 가능성을 크게 줄일 수 있다.
- dirty worktree에서 freeze bundle을 실제 staged candidate로 만들 때는 working tree 기준 확인만으로는 부족하고, 반드시 cached diff 기준으로 다시 검증해야 한다. exact inventory로 `git add` 한 뒤 `git diff --cached --name-only`, `git diff --cached --stat`, excluded-path grep, 그리고 대표 synthetic/browser gate를 staged 상태에서 다시 돌려야 commit approval 직전에 scope drift를 가장 확실하게 막을 수 있다.
- freeze bundle을 commit/push/PR까지 올린 뒤에도 같은 주제의 stale smoke literal이 남아 있으면, 다음 unrelated branch에서 그냥 들고 가지 말고 원래 bundle branch로 돌아가 smoke expectation drift만 따로 정리하는 편이 맞다. 이번처럼 `ui/app.js` 는 이미 noun-first CTA로 freeze됐는데 일부 `smoke-ui-slice-*` 만 예전 literal(`... 열기`, `... 만들기`)을 계속 기대하고 있었다면, runtime/provider 작업에 섞지 말고 원래 `company-shell` branch에서 smoke sync + reverify 후 같은 PR에 붙이는 편이 scope를 가장 덜 흐린다.
- source-of-truth 문서가 실제 product direction보다 뒤처져 dirty carryover로 남아 있으면, unrelated branch로 끌고 가지 말고 원래 기능 branch에서 닫는 편이 맞다. 이번처럼 `AGENTS.md` 가 이미 accepted 된 company-shell / `DESIGN.md` reality를 늦게 반영하고 있었다면, 별도 architecture work로 과장하지 말고 original company-shell branch에서 ledger + hygiene만 곁들여 sync 하는 편이 PR scope와 reviewer context를 가장 덜 흐린다.
- "하네스가 기본" 방향을 적용할 때는 runtime 코어를 늘리기보다 harness boundary를 문서와 smoke로 먼저 고정하는 게 안전했다. 이번처럼 `docs/13_harness-baseline.md` 로 정의를 고정하고, `markitdown` 같은 optional CLI wrapper를 독립 스크립트로 두고, `smoke-harness-slice-01` 로 존재/키워드를 핀하면 실제 기능 확장은 뒤로 미뤄도 방향성과 scope guard가 먼저 닫힌다.
- 하네스 레퍼런스를 볼 때 multi-provider, messaging gateway, guardrail 제거, `curl | bash` 설치 흐름은 v1 기본값으로 들이지 말아야 한다. 우리 레포는 local-first/optional harness만 허용하므로, 이런 요소는 "signal only"로 문서화하고 adoption 체크리스트에서 명시적으로 차단하는 편이 안정적이었다.
- 하네스 적용면을 늘리기 전에 repo-native inventory script로 `approved-now / future-post-v1 / signal-only` posture와 로컬 설치 여부를 먼저 보여 주는 편이 좋았다. availability와 adoption을 분리해 두면 "내 컴퓨터에 깔려 있으니 써도 된다" 같은 암묵적 scope drift를 막기 쉬웠다.
- inventory 다음 단계에서는 실행 entrypoint도 같은 posture 규칙을 강제하는 편이 맞았다. status 문서만 두면 signal-only 도구가 로컬에 설치된 순간 ad-hoc 실행이 다시 열릴 수 있으므로, 이번처럼 `harness-run` 이 `approved-now` 외 harness id를 명시적으로 거부하게 만들어 두는 편이 repo boundary를 더 단단하게 유지했다.
- approved-only entrypoint를 만든 다음에는 `list` 같은 self-describing subcommand를 바로 붙이는 편이 좋았다. 실행 가능한 harness를 별도 문서나 코드 열람 없이 바로 확인할 수 있어야 operator가 repo-native path를 쓰고, ad-hoc command guessing으로 다시 boundary 바깥으로 새지 않는다.
- 하네스 thin slice가 `status / policy gate / dispatch proof`로 늘어나기 시작하면, 개별 smoke를 사람이 기억해서 돌리는 대신 repo-native verification bundle로 바로 묶는 편이 좋았다. 그래야 harness 영역을 독립된 synthetic gate로 다시 확인할 수 있고, broader runtime/UI verification과도 경계를 섞지 않는다.
- `list` 다음 단계에서는 `info <harness-id>` 같은 per-harness 설명 entrypoint를 바로 붙이는 편이 좋았다. operator가 특정 harness의 posture, runner 유무, 설치 검토 지침을 docs 대신 repo-native command로 바로 볼 수 있어야, 승인 경계를 지키면서도 도구를 실제로 쓰기 쉬워진다.
- per-harness `info`를 붙인 다음에는 availability truth도 같은 helper로 맞춰 두는 편이 좋았다. status와 info가 서로 다른 probe 로직을 가지면 설치 상태가 어긋날 수 있으므로, 이번처럼 공통 probe helper 하나로 `available`과 `executable`을 같이 계산하게 두는 편이 drift를 줄였다.
- harness verification bundle을 한 번 만들고 나면 그 자체도 stale해지기 쉽다. info/status/policy slice가 추가될 때마다 verification bundle도 바로 최신 smoke set을 따라오게 업데이트해야, synthetic gate가 실제 repo truth보다 뒤처지는 문제를 막을 수 있다.
- verification bundle을 검증하는 smoke는 bundle 내부에 다시 넣으면 안 된다. 이번처럼 self-checking smoke가 recursion을 만들 수 있으므로, bundle contract smoke는 bundle 바깥에 두고 최신 count/id set만 따로 고정하는 편이 안전하다.
- harness operator ergonomics를 더 올릴 때는 `doctor` 같은 host summary가 중요했다. list/info/status가 각각 의미는 있지만, 실제 사용자는 지금 이 머신에서 무엇이 바로 되고 무엇이 설치 필요인지 먼저 알고 싶기 때문에, 공통 state helper 위에 `ready / install-required / deferred / policy-blocked`를 한 번에 보여 주는 command가 더 실용적이었다.
- `doctor` summary를 붙인 다음에는 action queue도 바로 같이 노출하는 편이 좋았다. counts와 full harness list만으로도 정보는 충분하지만, 실제 operator는 install-required를 먼저 보고 그다음 deferred와 policy-blocked를 구분해 판단하므로, top-level ordered queue 하나로 지금 해야 할 결정을 바로 보여 주는 편이 더 실용적이었다.
- action queue까지 붙인 다음에는 top-level `nextAction`도 같이 두는 편이 좋았다. 실제 operator는 queue 전체보다 "지금 첫 번째로 뭘 해야 하는가"를 먼저 읽기 때문에, 첫 item을 별도 필드로 끌어올려 두면 consumer가 queue parsing 없이 immediate decision을 바로 사용할 수 있었다.
- `nextAction`을 올린 다음에는 ready harness도 top-level로 따로 빼는 편이 좋았다. 현재 host에서 바로 실행 가능한 approved harness가 있는지 확인하려고 full harness list를 다시 스캔하게 두기보다, `readyHarnessIds` 하나로 runnable set을 바로 노출하는 편이 operator와 consumer 양쪽 모두에서 더 실용적이었다.
- runnable set을 top-level로 올린 다음에는 install-required set도 같이 분리하는 편이 좋았다. 현재 host에서 무엇을 설치해야 바로 다음 실행으로 넘어갈 수 있는지 확인하려고 actionQueue 전체를 다시 읽게 두기보다, `installRequiredHarnessIds` 하나로 local setup blocker를 바로 노출하는 편이 더 실용적이었다.
- local setup blocker를 top-level로 올린 다음에는 deferred set도 같이 분리하는 편이 좋았다. future-post-v1 direction을 확인하려고 full harness list나 actionQueue를 다시 스캔하게 두기보다, `deferredHarnessIds` 하나로 postponed set을 바로 노출하는 편이 operator와 consumer 양쪽 모두에서 더 실용적이었다.
- future-post-v1 set을 top-level로 올린 다음에는 signal-only blocked set도 같이 분리하는 편이 좋았다. adoption 금지 집합을 확인하려고 full harness list를 다시 스캔하게 두기보다, `policyBlockedHarnessIds` 하나로 blocked set을 바로 노출하는 편이 operator와 consumer 양쪽 모두에서 더 실용적이었다.
- top-level arrays를 다 올린 다음에는 compact summary도 같이 두는 편이 좋았다. consumer가 counts, booleans, next action state를 다시 조합하지 않게 `summary` 하나로 현재 host posture를 읽게 두면 상위 UI나 automation 쪽에서 payload 소비가 훨씬 단순해졌다.

- representative harness id를 붙인 다음에는 state도 같이 두는 편이 좋았다. consumer가 대표 id를 받은 뒤 다시 전체 payload에서 state를 lookup 하지 않게 `primaryHarnessState`까지 summary 안에 같이 두면 상위 UI/automation 구현이 더 단순해졌다.
- representative harness state를 붙인 다음에는 action도 같이 두는 편이 좋았다. consumer가 대표 harness의 현재 조치를 얻기 위해 다시 nextAction이나 actionQueue를 해석하지 않게 `primaryRecommendedAction`까지 summary 안에 같이 두면 상위 UI/automation 구현이 더 단순해졌다.
- representative harness action을 붙인 다음에는 install guidance도 같이 두는 편이 좋았다. consumer가 대표 harness 설치 지침을 얻기 위해 다시 harness list를 lookup 하지 않게 `primaryInstallReview`까지 summary 안에 같이 두면 상위 UI/automation 구현이 더 단순해졌다.
- representative harness install guidance를 붙인 다음에는 posture note도 같이 두는 편이 좋았다. consumer가 대표 harness 설명을 얻기 위해 다시 harness list를 lookup 하지 않게 `primaryNote`까지 summary 안에 같이 두면 상위 UI/automation 구현이 더 단순해졌다.
- representative harness note를 붙인 다음에는 posture bucket도 같이 두는 편이 좋았다. consumer가 대표 harness의 policy bucket을 얻기 위해 다시 harness list를 lookup 하지 않게 `primaryPosture`까지 summary 안에 같이 두면 상위 UI/automation 구현이 더 단순해졌다.
- representative harness posture를 붙인 다음에는 local command name도 같이 두는 편이 좋았다. consumer가 대표 harness의 실행 커맨드를 얻기 위해 다시 harness list를 lookup 하지 않게 `primaryCommand`까지 summary 안에 같이 두면 상위 UI/automation 구현이 더 단순해졌다.
- representative harness command를 붙인 다음에는 repo-native runner path도 같이 두는 편이 좋았다. consumer가 대표 harness의 wrapper 진입점을 얻기 위해 다시 harness list를 lookup 하지 않게 `primaryRunner`까지 summary 안에 같이 두면 상위 UI/automation 구현이 더 단순해졌다.
- representative harness runner를 붙인 다음에는 executable truth도 같이 두는 편이 좋았다. consumer가 대표 harness가 현재 repo posture에서 실제 실행 가능한지 알기 위해 다시 harness list를 lookup 하지 않게 `primaryExecutable`까지 summary 안에 같이 두면 상위 UI/automation 구현이 더 단순해졌다.
- representative harness executable truth를 붙인 다음에는 host PATH availability truth도 같이 두는 편이 좋았다. consumer가 대표 harness command가 현재 host에서 실제로 감지되는지 알기 위해 다시 harness list를 lookup 하지 않게 `primaryAvailable`까지 summary 안에 같이 두면 repo-native gate truth와 PATH detection truth를 나란히 읽기 쉬워졌다.
- representative harness availability truth를 붙인 다음에는 harness kind classification도 같이 두는 편이 좋았다. consumer가 대표 harness가 어떤 분류의 local helper인지 알기 위해 다시 harness list를 lookup 하지 않게 `primaryKind`까지 summary 안에 같이 두면 operator 설명과 상위 UI copy를 더 직접적으로 만들 수 있었다.
- representative harness kind를 붙인 다음에는 install-review requirement를 같이 두는 편이 좋았다. consumer가 대표 harness의 installReview 유무를 다시 harness list에서 찾지 않게 `primaryInstallReviewRequired`를 summary에 올리면 setup 판단과 operator 안내를 즉시 분리해 읽을 수 있었다.
- representative harness install-review requirement를 붙인 다음에는 ready 여부도 같이 두는 편이 좋았다. consumer가 대표 harness가 지금 바로 runnable인지 확인하기 위해 다시 harness list를 lookup 하지 않게 `primaryReady`까지 summary에 올리면 상태 확인이 더 직접적으로 닫힌다.
- representative harness ready 여부를 붙인 다음에는 action shorthand도 같이 두는 편이 좋았다. consumer가 대표 harness에 대해 현재 어떤 조치가 필요한지 간단 라벨로 바로 읽게 `primaryActionShort`를 summary에 올리면 UI 문구와 operator 안내가 더 즉시화된다.
- representative harness action shorthand를 붙인 다음에는 operator-facing 한 줄 guidance도 같이 두는 편이 좋았다. consumer가 `install` 같은 short label을 다시 사람 친화적 문장으로 바꾸지 않게 `primaryActionMessage`를 summary에 올리면 상위 UI와 automation에서 안내 문구를 더 직접적으로 재사용할 수 있다.
- representative summary는 `primaryActionMessage` 수준에서 멈추는 편이 좋았다. 이 지점 이후의 summary field 확장은 contract hardening이 아니라 consumer preference가 되기 쉬우므로, 추가 필드는 별도 milestone 또는 consumer-driven justification이 있을 때만 다시 여는 편이 scope drift를 막는다.
- contract를 freeze한 뒤 첫 consumer를 붙일 때는 producer에 field를 더 추가하지 말고, frozen `doctor.summary`만 읽는 별도 consumer surface를 만드는 편이 좋았다. consumer가 raw `harnesses[]`나 top-level arrays를 다시 조합하기 시작하면 freeze 의미가 약해지므로, compact summary를 source-of-truth로 두고 consumer payload를 더 작은 `statusCard`로 재가공하는 편이 boundary를 지키기 쉽다.
- approved-now harness가 실제로 `ready`가 된 순간에는 `actionQueue`, `nextAction`, representative summary가 모두 그 runnable approved harness를 먼저 가리키게 두는 편이 맞았다. 그렇지 않으면 current host posture는 `runnable`인데도 representative harness가 `deferred` 쪽으로 밀려 operator surface가 실제 실행 경로와 어긋난다.
- install review가 `pipx` 중심으로 적혀 있더라도 host에 `pipx`가 없고 이미 active local Python toolchain이 잡혀 있다면, host-ready proof는 그 Python에 직접 설치하는 편이 더 작은 batch였다. repo guidance는 review-first로 유지하되, 단지 preferred installer가 없다는 이유로 setup scope를 불필요하게 넓히지 않는 편이 낫다.
- frozen producer contract 위에 consumer를 더 붙일 때는 `doctor.summary`에 field를 다시 추가하기보다 consumer가 이미 닫힌 summary에서 바로 파생 가능한 operator surface를 별도 payload로 내는 편이 좋았다. 이번처럼 repo-native run command를 consumer `operatorAction`으로 노출하면 producer freeze를 유지하면서도 실제 실행 UX는 더 직접적으로 닫을 수 있다.
- post-freeze consumer가 두 단계 이상 필요할 때는 다시 doctor를 읽지 말고 바로 앞 consumer payload를 읽는 편이 layering이 더 명확했다. 이번처럼 `doctor.summary -> consumer status -> consumer brief` 순서로 좁혀 가면 shell-friendly surface를 추가하면서도 source-of-truth 경계가 흐려지지 않는다.
- post-freeze consumer를 실제 UI에 연결할 때도 browser에서 harness script를 직접 실행하게 하거나 `doctor` producer를 다시 열지 않는 편이 맞았다. 이번처럼 `doctor.summary -> consumer status -> consumer brief -> snapshot derived -> ui signal` 순서로 bridge를 하나 더 두면, UI는 기존 `/api/snapshot` 경로만 읽으면서도 frozen harness contract를 안전하게 consume할 수 있었다.
- frozen brief가 이미 snapshot에 bridge되어 있다면, 다음 UI step은 새 global strip을 더 추가하기보다 기존 panel 안에 read-only register 하나를 여는 편이 맞았다. 이번처럼 `Ops overview` org panel에 `headline / action / command` register를 붙이면 operator가 같은 viewport에서 바로 다음 실행을 읽을 수 있고, UI consumer도 계속 `derived.harnessConsumerBrief`만 읽게 되어 producer boundary가 흐려지지 않았다.
- frozen brief register가 한 surface에서 검증됐다면, 다음 UI 확장은 새 payload를 만들기보다 같은 register helper를 다른 ops surface에 재사용하는 편이 더 안전했다. 이번처럼 `Taskboard`에 같은 read-only register를 한 번 더 배치하니, consumer layering은 그대로 유지하면서 실제 operator surface coverage만 넓힐 수 있었다.
- frozen brief register가 `Taskboard`까지 검증됐다면, 다음 advanced-ops 확장도 새 panel semantics를 만들기보다 같은 helper를 `Logs` 같은 기존 room deck 바로 아래에 재사용하는 편이 더 안전했다. consumer payload를 늘리지 않고 insertion point만 `deck -> read-only register -> existing surface grid`로 고정하면, frozen brief layering을 유지하면서도 operator coverage를 한 단계 더 넓힐 수 있었다.
- 같은 frozen brief register를 여러 advanced-ops room에 늘릴 때도 매번 새 변형을 만들기보다 `Logs`, `Artifacts`처럼 각 room의 첫 deck 바로 아래에 같은 helper를 반복 배치하는 편이 더 안전했다. insertion point만 `deck -> read-only register -> existing surface grid`로 통일해 두면 smoke도 단순해지고, consumer contract를 넓히지 않고도 room coverage를 예측 가능하게 확장할 수 있었다.
- room-level harness consumer 확장을 끝까지 닫을 때도 `Decision Inbox`만 예외 취급하지 않는 편이 좋았다. 마지막 room도 같은 `deck -> read-only register -> existing surface grid` 패턴으로 맞추면 advanced-ops 전체가 동일한 frozen brief entry rhythm을 공유하고, 각 room smoke도 insertion point 하나만 다르게 고정하면 되므로 유지 비용이 낮다.
- room-level harness consumer 확장을 마친 뒤 다음 step은 새 room을 더 만드는 게 아니라 실행에 가장 가까운 shell surface로 내려오는 편이 맞았다. 이번처럼 `Execution`에서는 `renderExecutionCommandDeck` 바로 아래에 같은 read-only register를 붙이면 frozen brief가 실제 run CTA 문맥에 붙으면서도 interactive semantics를 새로 만들지 않고 `deck -> register -> work-order grid` 리듬을 유지할 수 있었다.
- execution-adjacent shell surface로 내려온 뒤 다음 step은 그와 짝을 이루는 packet handoff surface를 같은 패턴으로 닫는 편이 맞았다. 이번처럼 `Deliverables`에서는 `deliverablesDeck` 바로 아래에 같은 read-only register를 붙이면 frozen brief가 결과 패킷 문맥까지 이어지면서도 producer나 snapshot semantics를 넓히지 않고 `deck -> register -> surface grid` 리듬을 유지할 수 있었다.
- room-level brief coverage가 끝난 뒤 첫 interactive step은 새 runtime route나 shell launch API를 여는 게 아니라 shared register helper에 local-only action affordance 하나를 붙이는 편이 맞았다. 이번처럼 `brief.actionCommand`만 이용한 `명령 복사` 버튼과 clipboard-or-status fallback을 `renderHarnessBriefRegister()`에 넣으면 producer freeze와 snapshot layering을 그대로 유지하면서도 operator ergonomics를 한 단계 올릴 수 있었다.
- shared register에 첫 local-only action이 붙은 다음 단계도 새 shell launch API를 여는 것보다 existing surface navigation을 재사용하는 편이 맞았다. 이번처럼 `실행 데스크 열기`를 `open-surface -> execution`으로 연결하면 producer/snapshot/runtime 경계를 그대로 지키면서도 operator가 canonical run surface로 바로 이동할 수 있어 interactive 흐름을 가장 작은 배치로 넓힐 수 있었다.
- shared brief register에 local-only action이 붙은 뒤 execution-facing next step은 brief에 필드를 더 얹는 것보다 frozen `harnessConsumerStatus.operatorAction`을 별도 shelf로 브리지하는 편이 맞았다. 이번처럼 `Execution` 안에서만 전용 `하네스 실행 액션` shelf를 두면 `brief`는 cross-surface register로 유지하고, 실제 run/install/deferred kind는 execution 문맥에서만 더 구체적으로 읽게 분리할 수 있어 consumer layering이 더 명확해진다.
- ready harness가 존재하더라도 wrapper가 runtime argument를 요구하면 consumer surface는 bare command를 runnable truth처럼 노출하면 안 됐다. 이번처럼 `markitdown` bare dispatch가 usage를 반환한다는 기존 smoke를 근거로 `primaryActionMessage`, `operatorAction.repoNativeCommand`, `brief.actionCommand`를 required-args template(`node scripts/harness-run.mjs markitdown <input-file> [output-file]`)로 바로잡는 편이 다음 interactive launch step보다 먼저여야 했다.
- third-party CLI host-ready smoke는 mutable repo 문서나 localized content를 직접 fixture로 쓰지 않는 편이 안전했다. 이번처럼 `markitdown` 0.1.5가 비-ASCII 입력에서 `ascii` decode failure를 내는 순간 `smoke-harness-slice-34`가 unrelated doc drift에 묶였기 때문에, host-ready gate는 tempDir 안의 ASCII-only fixture를 직접 만들고 그 위에서 repo-native wrapper만 증명하는 편이 훨씬 안정적이었다.
- explicit execution mutation path를 열 때도 generic shell exec surface부터 만들면 scope가 바로 무너진다. 이번처럼 `Execution` 안의 frozen `operatorAction.kind === repo-native-run`만 신뢰하고, 현재 approved representative harness(`markitdown`)에 한정된 `POST /api/harness/operator-action/run` route로 좁힌 뒤 입력/출력 경로만 받게 하면 local-only interactive execution을 열면서도 producer, snapshot, shell-launch boundary를 그대로 유지할 수 있었다.
