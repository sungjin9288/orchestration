## Implementation Plan

The later implementation slice should add durable proposal record creation and persistence as one narrow local-first path.

The implementation should:

- extend the runtime state contract with a `proposalRecords` collection and a `proposalRecord` sequence
- normalize missing `proposalRecords` and `proposalRecord` sequence values when existing runtime state is loaded
- create proposal records only through one explicit creation function that requires an approved creation payload
- persist records in the existing runtime `state.json` under the selected runtime root
- use stable ids in the existing sequence style, such as `proposal-record-0001`
- set initial status to `created`
- set `createdAt` and `updatedAt` from the runtime clock at creation time
- set `expiresAt` from the approved creation payload or a documented default in the implementation slice
- copy source evidence refs, negative evidence refs, reviewer refs, approval refs, risk class, blocked actions, and verification plan into the record
- force `applyAllowed` to `false` for every created record
- expose records as read data only on the existing proposal review surface

The implementation must not:

- generate proposals
- apply proposals
- approve proposal application
- mutate proposal queues outside the approved record creation path
- call providers
- persist memory
- ingest raw transcripts
- promote skills
- mutate source from growth candidates
- commit
- push

## Record Contract

The implementation should create records with this minimum shape:

| Field | Rule |
| --- | --- |
| `proposalId` | Stable runtime id generated by the approved creation path. |
| `title` | Required non-empty operator-facing title. |
| `proposalType` | Required value from the existing proposal queue vocabulary. |
| `status` | Initial value `created`. |
| `createdAt` | Runtime timestamp assigned at creation. |
| `updatedAt` | Same as `createdAt` on creation. |
| `expiresAt` | Required timestamp or documented default from the approved implementation slice. |
| `sourceClaimIds` | Required non-empty Growth Evidence Ledger claim refs. |
| `evidenceRefs` | Required non-empty positive evidence refs. |
| `negativeEvidenceRefs` | Required non-empty risk, missing-proof, or blocker refs. |
| `reviewerRefs` | Required non-empty reviewer or review-evidence refs. |
| `approvalRefs` | Required non-empty creation approval refs. |
| `affectedFiles` | Repo-relative files or docs affected by the proposed future change. |
| `riskClass` | Existing queue risk class. |
| `approvalGate` | Creation approval only. |
| `reviewQuestion` | Human review question preserved from the candidate. |
| `verificationPlan` | Focused and aggregate checks required for this record. |
| `blockedActions` | Explicit list including proposal application, provider calls, memory persistence, source mutation, commit, and push. |
| `applyAllowed` | Always `false`. |
| `nonApprovalStatement` | Statement that creation is not proposal approval or application approval. |

## Rollback Plan

The later implementation slice must include rollback evidence for these steps:

1. Disable the proposal record creation entrypoint.
2. Restore the previous runtime `state.json` shape when no records existed before the slice.
3. Preserve existing runtime projects, missions, tasks, runs, artifacts, approvals, and decision inbox items.
4. Quarantine invalid local proposal records instead of silently deleting audit evidence.
5. Prove that proposal application, provider calls, memory persistence, source mutation, commit, and push stay blocked after rollback.
6. Run focused smoke checks and aggregate verification after rollback.

## Focused Smoke Plan

The later implementation slice must add focused smoke coverage proving:

- record creation fails without explicit creation approval
- record creation fails when source evidence refs are missing
- record creation fails when negative evidence refs are missing
- record creation fails when reviewer refs are missing
- created records include stable id, status, timestamps, expiry, source evidence refs, negative evidence refs, reviewer refs, approval refs, risk class, blocked actions, verification plan, and non-approval statement
- `applyAllowed` remains `false`
- proposal application remains blocked
- provider calls remain blocked
- memory persistence remains blocked
- source mutation remains blocked
- commit and push remain blocked
- aggregate verification includes the new focused smoke

## Stop Conditions

Stop before implementation when any of these are true:

- no later `approve-implementation-slice` decision exists
- the implementation decision names more than one authority path
- the storage path is not the existing runtime `state.json`
- id, timestamp, expiry, rollback, or smoke policy is missing
- proposal application is coupled to record creation
- provider calls, memory persistence, source mutation, commit, or push would open by implication
- rollback evidence is missing
- aggregate verification is not registered
