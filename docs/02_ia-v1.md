# Orchestration 1.0 IA v1

## Purpose
This document defines the information architecture for the first usable Orchestration 1.0 product. It maps the primary surfaces, the core objects, and the minimum user flows needed for `development pack only`.

## IA Principles
- optimize for operational visibility, not decorative navigation
- keep the primary path centered on project, task, run, review, and approval
- make blockers and gates visible where the operator works
- keep object relationships inspectable from any primary surface
- treat chat as optional assistance, not the product shell

## Primary Navigation
V1 has four top-level destinations:

1. `Taskboard`
2. `Logs`
3. `Artifacts`
4. `Decision Inbox`

The navigation is scoped by the active project. Project switching is global because `project_path` is required before any execution.

## Global Shell
### Global Header
- active project name
- resolved `project_path`
- runtime root
- active run count
- pending decision count

### Global Actions
- The current shell keeps most actions inside the active surface instead of a persistent header action strip.
- `Taskboard` owns project registration and selection, linked worktree create and switch, task creation, and the current execution action stack.
- `Artifacts` and `Decision Inbox` surface approval follow-up without forcing a surface change.

### Cross-Surface Context
The current project, selected task, latest run, and pending gate state should remain easy to recover when moving between surfaces.

## Core Objects

### Project
- owns `project_path`
- scopes tasks, runs, worktrees, artifacts, decisions, reviews, and approvals

### Task
- contains title, intent, state, blocker flags, linked worktree, and linked outputs

### Run
- contains execution status, role, timestamps, exit result, and live or retained logs

### Worktree
- contains repo isolation metadata and merge or discard outcome

### Artifact
- contains type, source run or task, raw storage reference, and derived preview or browse policy

### Decision
- contains gate type, prompt, owner, due state, and resolution

### Review
- contains findings, verification evidence, and pass or request-changes outcome

### Approval
- contains approver, scope, decision timestamp, and allowed next action

## Primary Surfaces

### 1. Taskboard
The main operating surface for task state and progress.

#### Shows
- first-run bootstrap and project registry
- detected linked worktrees for the active project
- task columns or grouped lists by lifecycle state
- blocker and pending-gate badges
- latest run summary
- linked worktree status
- linked artifact count

#### Primary Actions
- register or select project
- create linked worktree and switch active project to a linked worktree root
- create task
- run planner, architect, task-breaker, and builder preflight
- request builder live-mutation approval and run bounded live mutation
- run reviewer, prepare commit-package, run local commit, prepare release-package, and close out when guards allow
- open task detail and preselected gate context

#### Required Detail Panel
- task summary and intent
- latest run
- task/worktree relation against the current active project
- linked artifacts
- review status
- approval requirement
- decision history
- runtime-derived readiness summaries for builder live mutation, reviewer, commit-package, local commit, release-package, and close-out

### 2. Logs
The run-level execution surface.

#### Shows
- live terminal output
- historical runs by task
- last failure and last successful checkpoint
- role or executor metadata
- related task and artifact linkage

#### Primary Actions
- inspect current run
- switch between runs
- open related task
- open preferred artifacts generated by the run
- follow provenance into review, commit, release, or close-out evidence

### 3. Artifacts
The evidence and output surface.

#### Shows
- artifacts grouped by project, task, and type
- previews where possible, with best-effort structured views and raw fallback
- origin run and task
- provenance linkage across builder, reviewer, commit, release, and close-out bundles
- latest generated outputs first, while `plan`, `architecture`, and `breakdown` remain latest-centered browse with retained history

#### Primary Actions
- preview artifact
- jump to source task or run
- follow commit or release provenance without leaving the surface
- approve or reject the current preselected approval when the selected artifact exposes that action
- inspect current readiness summaries for builder live mutation, commit-package, release-package, and close-out

### 4. Decision Inbox
The gate management surface for human-required actions.

#### Shows
- review gate markers, decision follow-up, and approval requests in one queue
- `review-follow-up` normalized as `kind=decision, sourceType=review`
- `unblock/clarification` normalized as `kind=decision, sourceType=decision, blocksTask=false`
- age, owner, and severity
- affected task and project
- next recommended action

#### Primary Actions
- approve
- reject
- resolve
- inspect related task, approval, and readiness context
- keep `review` items read-only and resolve them through the review flow instead of the generic inbox action route

## Default Task Lifecycle View
V1 should start with a simple lifecycle and expose blocking conditions as visible badges:

1. `Inbox`
2. `In Progress`
3. `Review`
4. `Done`

`Blocked`, `Waiting Approval`, and `Waiting Decision` should be visible from every relevant surface even if they remain flags instead of first-class columns.

## Core User Flows

### Flow A: Start Work
1. Register or select a project with a valid `project_path` on `Taskboard`.
2. Create or open a task on `Taskboard`.
3. Run `planner -> architect -> task-breaker -> builder preflight`.
4. Request live-mutation approval for the latest preflight, then run bounded live mutation.
5. Follow runs and logs on `Logs`.

### Flow B: Inspect Evidence
1. Open a task or run.
2. Jump to `Artifacts`.
3. Review generated outputs and provenance for the latest builder, reviewer, commit, release, or close-out bundle.
4. Use the current artifact detail view to follow approval or close-out readiness without leaving `Artifacts`.

### Flow C: Resolve A Gate
1. Open `Decision Inbox`.
2. Review the blocking item and its task context.
3. approve or reject approval items, or resolve decision items
4. return to `Taskboard`, `Artifacts`, or `Decision Inbox` with the updated readiness state

### Flow D: Close Work
1. Run reviewer so the task enters `Review` with explicit review evidence.
2. Prepare `commit-package`, resolve commit approval, and run local commit.
3. Prepare `release-package` from the latest successful local commit bundle on the dedicated linked worktree root when required.
4. Resolve the `release-ready` approval and run `close-out`.
5. Transition `Review -> Done` only after the current approved release bundle and repo-clean guard both pass.

## Empty, Error, And Blocked States
- no project selected: prompt for project selection and explain that `project_path` is required
- no tasks: suggest creating the first thin-slice task
- no runs yet: explain how to start execution from `Taskboard`
- no artifacts: explain that artifacts appear after execution or review
- blocked task: show exact blocker, owner, and next action
- run or guard failure: keep the current project and task visible, show the coordinator or runtime reason directly, and avoid inferring replacement semantics in the client

## Out Of Scope For IA v1
- office view or radar map as primary navigation
- messenger inboxes as first-class surfaces
- ranking dashboards
- pack marketplace
- multi-provider administration UI
