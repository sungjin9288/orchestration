# Proposal Application Implementation Plan

## Purpose

This document records the accepted `approve-application-planning-only` operator decision and the implementation plan for proposal application semantics on existing durable proposal records.

It is not proposal application implementation approval. It does not apply proposals, generate proposals, call providers, persist memory, mutate source, commit, or push.

## Application Plan

The first implementation slice should add one inert proposal application lifecycle path for existing durable proposal records.

The implementation should:

- accept only existing records from the approved local runtime `proposalRecords` collection
- require `status=created`
- require `applyAllowed=false` before the attempt starts
- require non-empty `sourceClaimIds`, `evidenceRefs`, `negativeEvidenceRefs`, `reviewerRefs`, `approvalRefs`, and `verificationPlan`
- require an explicit application approval payload separate from the creation approval payload
- reject expired, superseded, quarantined, stale, or already-applied records
- record an application attempt as audit evidence before any later source mutation authority can be considered
- keep the first application path documentation-only and audit-only unless a later implementation decision names a narrower mutation path
- keep provider calls, memory persistence, source mutation, commit, and push blocked
- preserve the original proposal record and append attempt evidence instead of overwriting source evidence

The implementation must not:

- generate proposals
- apply source changes
- mutate files from a proposal
- call providers
- persist memory
- ingest raw transcripts
- promote skills
- commit
- push
- treat proposal record creation approval as application approval
- treat application approval as source mutation approval

## Application Contract

The first implementation slice should model application as an attempt record, not as direct source mutation.

| Field | Rule |

## Rollback Plan

The later implementation slice must include rollback evidence for these steps:

1. Disable the proposal application entrypoint.
2. Preserve existing durable proposal records and their creation audit evidence.
3. Quarantine failed or invalid application attempts instead of deleting them silently.
4. Restore the previous runtime state shape when no application attempts existed before the slice.
5. Prove that provider calls, memory persistence, source mutation, commit, and push stay blocked after rollback.
6. Run focused smoke checks and aggregate verification after rollback.

## Focused Smoke Plan

The later implementation slice must add focused smoke coverage proving:

- application planning is accepted but implementation remains blocked until a later decision
- application fails without explicit application approval
- application fails when the referenced proposal record does not exist
- application fails when the proposal record is expired, superseded, quarantined, stale, or already applied
- application fails when source evidence refs are missing
- application fails when negative evidence refs are missing
- application fails when reviewer refs or creation approval refs are missing
- application attempt evidence preserves the original proposal record
- source mutation remains blocked
- provider calls remain blocked
- memory persistence remains blocked
- commit and push remain blocked
- aggregate verification includes the new focused smoke

## Implementation Decision Required

Before implementation can start, the operator must provide a later `approve-application-implementation-slice` decision that names exactly one application path and points back to this document.

That later decision must keep source mutation, provider calls, memory persistence, commit, and push separate unless those actions are separately approved.

## Stop Conditions

Stop before implementation when any of these are true:

- no later `approve-application-implementation-slice` decision exists
- the implementation decision names more than one authority path
- application approval is collapsed into proposal record creation approval
- application approval is collapsed into source mutation approval
- application attempts overwrite proposal record source evidence
- expired, superseded, quarantined, stale, or already-applied records are not rejected
- rollback evidence is missing
- focused smoke refs are missing
- aggregate verification is not registered
- provider calls, memory persistence, source mutation, commit, or push would open by implication
